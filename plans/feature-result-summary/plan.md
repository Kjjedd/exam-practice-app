# feature/result-summary Plan

## Goal

세션이 끝난 뒤 결과 화면에서 전체 문제 수, 정답 수, 오답 수, 정답률, 그리고 문제별 정답/오답 구분을 확인할 수 있는 요약 흐름을 만든다.

## Branch Intent

이 브랜치는 "문제 이동이 끝난 뒤 어디로 가는가"를 "무엇을 요약해서 보여주는가"로 확장하는 단계다.

- [x] 세션 결과를 요약할 수 있는 최소 집계 구조를 만든다.
- [x] `/result`를 플레이스홀더가 아닌 실제 결과 요약 화면으로 바꾼다.
- [x] 문제별 정답/오답 상태를 결과 화면에서 구분해 보여준다.
- [x] 다음 단계인 오답 복습 기능이 자연스럽게 올라갈 수 있도록 오답 문제 식별 기준을 분명히 유지한다.

이번 단계의 핵심은 "결과를 계산하고 보여주는 것"이지, 오답만 다시 푸는 흐름을 구현하는 것이 아니다.

## Scope

- 세션 결과 집계 구조 도입
- 총 문제 수 계산
- 정답 수 계산
- 오답 수 계산
- 정답률 계산
- 문제별 정답/오답 상태 요약 목록
- 결과 화면 UI
- 세션 종료 시 결과 화면으로 필요한 데이터 전달 또는 복원

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 현재 퀴즈 화면에서 문제별 제출 결과를 누적할 수 있는 최소 구조 정의
- 마지막 문제 종료 시 결과 화면이 집계에 필요한 데이터를 받을 수 있게 연결
- `/result`에서 전체 세션 결과를 집계하고 시각적으로 요약
- 각 문제에 대해 정답/오답 여부를 한눈에 구분할 수 있는 목록 제공
- 결과 화면이 비어 있거나 입력이 부족할 때의 예외 상태도 정리

이번 브랜치에서는 결과를 "읽고 이해하는 화면"까지 다루지만, 그 결과를 저장하거나 복습 세션으로 재구성하는 단계까지는 포함하지 않는다.

## State Principles

결과 집계 상태는 아래 원칙을 따라야 한다.

- 결과는 세션 단위로 계산되어야 한다.
- 문제별 결과는 `questionId` 기준으로 추적 가능해야 한다.
- 전체 집계 값은 문제별 결과 배열에서 유도 가능해야 한다.
- 결과 화면은 실제 세션 결과가 없으면 안전한 빈 상태를 보여야 한다.
- 결과 집계는 현재 표시 순서와 무관하게 일관된 값을 반환해야 한다.

## Proposed Data Shape

이번 단계에서는 최소한 아래 수준의 구조가 필요할 가능성이 높다.

- `QuestionResult[]`
- `QuizSession`
- 유도 집계 값:
  - `totalQuestions: number`
  - `correctCount: number`
  - `wrongCount: number`
  - `accuracyRate: number`

필요하다면 결과 화면 전용 요약 타입을 별도로 둘 수 있다.  
핵심은 결과 계산 책임과 결과 표시 책임을 적절히 분리하는 것이다.

## Aggregation Rules

결과 집계는 최소한 아래 규칙을 만족해야 한다.

- 총 문제 수는 세션에 포함된 문제 수를 기준으로 계산한다.
- 정답 수는 `isCorrect === true`인 결과 수를 기준으로 계산한다.
- 오답 수는 총 문제 수에서 정답 수를 뺀 값 또는 `isCorrect === false` 기준으로 계산한다.
- 정답률은 `정답 수 / 총 문제 수 * 100`으로 계산하되, 0문제 예외를 안전하게 처리해야 한다.
- 문제별 결과 목록은 현재 세션 문제와 결과를 연결할 수 있어야 한다.

## Interaction Design

이번 단계에서 기대하는 상호작용은 다음과 같다.

### 1. Session Complete

- 사용자가 마지막 문제를 마치고 결과 화면으로 이동한다.
- 결과 화면은 전체 세션 요약을 즉시 보여준다.

### 2. Overview First

- 사용자는 먼저 총 문제 수, 정답 수, 오답 수, 정답률 같은 요약 지표를 본다.
- 이 요약은 한 번에 읽히는 카드 또는 상단 섹션으로 배치되는 편이 좋다.

### 3. Per-question Breakdown

- 사용자는 각 문제가 정답이었는지 오답이었는지 확인할 수 있다.
- 문제별 상태는 이후 오답 복습으로 이어질 수 있도록 명확해야 한다.

### 4. Empty Or Invalid Result State

- 세션 결과가 없거나 직접 `/result`로 진입한 경우에는 안전한 안내 상태가 필요하다.
- 이 경우 홈 또는 퀴즈 시작으로 돌아갈 수 있는 안내가 있어야 한다.

### 5. Non-goals in This Step

- 오답만 다시 풀기 시작
- 즐겨찾기 저장
- 통계 대시보드 누적 반영
- 결과 영속 저장

## UI Principles

- 결과 화면은 "끝났다"는 느낌과 "어떻게 끝났는지"를 동시에 전달해야 한다.
- 숫자 요약과 문제별 목록은 시각적으로 분리되어야 한다.
- 정답/오답 상태는 색과 텍스트 둘 다로 구분되어야 한다.
- 결과가 비어 있을 때도 화면이 끊기지 않아야 한다.
- 이후 오답 복습 CTA가 올라갈 위치를 남겨둘 수 있도록 구조가 단순해야 한다.

## Result Display Rules

결과 화면은 최소한 아래 정보를 다뤄야 한다.

- 총 문제 수
- 정답 수
- 오답 수
- 정답률
- 문제별 정답/오답 상태

문제별 목록에는 적어도 문제 번호 또는 문제 식별 정보와 정답/오답 상태가 표시되어야 한다.

## Data Flow Notes

현재 구조상 결과 화면은 아직 세션 데이터를 직접 받지 않는다.  
이번 단계에서는 아래 중 한 방향으로 정리될 가능성이 높다.

- 퀴즈 화면에서 결과 집계에 필요한 세션 상태를 라우트 전이 시 전달
- 공유 가능한 세션 상태를 클라이언트 메모리 또는 브라우저 저장소에 잠시 보관

어느 방식을 택하든 이번 단계의 핵심은 "결과 화면이 세션 종료 직후 필요한 결과를 읽을 수 있게 만드는 것"이다.

## Out of Scope

- 오답 복습 세션 시작
- 랜덤 모드 전용 결과 차별화
- 시험 모드 전용 결과 차별화
- 누적 통계 저장
- 서버 전송
- 장기 보관용 결과 히스토리

## Risks

- 결과 집계와 오답 복습 로직을 한 번에 섞으면 다음 브랜치 경계가 무너진다.
- 현재 세션 상태 전달 방식이 불안정하면 `/result` 새로고침 시 화면이 깨질 수 있다.
- 문제별 결과와 실제 문제 배열 연결이 어긋나면 잘못된 정답/오답 표시가 생길 수 있다.
- 결과 화면에서 너무 많은 정보를 한 번에 넣으면 핵심 요약이 흐려질 수 있다.

## Notes

- 이번 브랜치의 핵심은 "마지막 문제 이후 사용자가 무엇을 배웠는지 요약해 보여주는 것"이다.
- 숫자 요약은 빠른 이해를 위해 중요하고, 문제별 목록은 다음 행동을 준비하기 위해 중요하다.
- 저장과 복습은 다음 단계로 넘기고, 이번 단계는 결과 집계의 신뢰도를 먼저 확보하는 편이 안전하다.

## Exam Template Result Revision

추가 수정에서는 시험 템플릿 기반 세션 결과를 더 구체적으로 보여줄 수 있어야 한다.

- 시험 모드 세션이면 선택한 시험 코드와 시험명을 결과 상단에서 확인할 수 있어야 한다.
- 총 문항 수는 시험 템플릿 기준 문항 수와 일치해야 한다.
- 채점 방식은 "공식 채점 설명"과 "앱 연습용 합격 판정"을 구분해서 보여주는 편이 안전하다.
- 결과 카드 또는 보조 패널에서 합격 기준과 현재 세션의 연습용 판정 결과를 보여줄 수 있어야 한다.
