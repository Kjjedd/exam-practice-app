# feature/question-ui Plan

## Goal

활성 문제 세트에서 문제 한 개를 안정적으로 보여주고, 문제 세트가 없을 때는 가져오기 흐름으로 안내하는 문제풀이 화면 구조를 만든다.

## Branch Intent

이 브랜치는 "퀴즈 페이지가 저장된 활성 문제 세트를 읽어 실제 문제 한 개를 안정적으로 표시하고, 데이터가 없을 때의 빈 상태도 다룬다"는 목적만 담당한다.

- [x] 활성 문제 세트를 받아 화면에 표시할 수 있는 기본 골격을 만든다.
- [x] 문제 번호, 본문, 보기 목록, 진행 상태를 명확하게 보여준다.
- [x] 활성 문제 세트가 없을 때 가져오기 흐름으로 안내하는 빈 상태를 만든다.
- [x] 다음 단계의 선택 로직과 채점 로직이 자연스럽게 올라갈 수 있는 UI 구조를 준비한다.

이 단계에서는 아직 선택 상태, 제출 버튼 동작, 정답 판정 같은 상호작용 로직은 구현하지 않는다.  
기존의 정적 샘플 문제 기반 화면은 초기 검증용 결과였으며, 현재 계획에서는 저장된 활성 문제 세트를 기준으로 다시 정의한다.

## Scope

- 문제 번호 표시
- 문제 본문 표시
- 보기 목록 렌더링
- 진행 상태 표시
- 활성 문제 세트 기반 실제 문제 데이터 연결
- 활성 문제 세트 없음 상태 처리
- 퀴즈 페이지 기본 레이아웃 구성
- 모바일/데스크톱 대응 구조 구성

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- `feature/question-data`에서 준비된 활성 문제 세트 데이터를 화면에 연결
- 퀴즈 페이지가 최소한 한 문제를 읽기 좋게 보여주도록 구조 설계
- 문제 카드와 보기 목록의 시각적 분리
- 현재 문제 위치를 알 수 있는 진행 정보 표시
- 활성 문제 세트가 비어 있거나 존재하지 않을 때의 안내 상태 설계
- 이후 `feature/answer-selection`에서 선택 상태만 얹으면 되도록 UI 구조 분리

이번 브랜치의 핵심은 "문제풀이 인터페이스의 읽기 구조"를 만드는 것이다.

## UI Principles

퀴즈 화면은 아래 원칙을 따라야 한다.

- 문제 본문이 보기보다 먼저 명확히 읽혀야 한다.
- 보기 목록은 한눈에 구분 가능해야 한다.
- 현재 몇 번째 문제를 보고 있는지 사용자가 알아야 한다.
- 긴 문제 텍스트와 긴 보기 텍스트가 있어도 레이아웃이 무너지지 않아야 한다.
- 문제 세트가 없을 때 사용자가 왜 문제를 볼 수 없는지 이해할 수 있어야 한다.
- 이후 선택 상태 강조가 추가되더라도 구조를 바꿀 필요가 없도록 설계한다.

## Information Structure

이번 단계에서 퀴즈 화면에 포함되면 좋은 구조는 다음과 같다.

### 1. Quiz Header

역할:

- 현재 문제 위치를 알려준다.
- 퀴즈 화면이라는 맥락을 보여준다.

포함 가능 정보:

- 전체 문제 수 대비 현재 번호
- 현재 모드 또는 기본 라벨
- 활성 문제 세트 이름

### 2. Progress Summary

역할:

- 현재 진행 상태를 간단히 시각화한다.

예시:

- `1 / 20`
- 진행 바 또는 진행 텍스트

이번 단계에서는 실제 정답률/채점 진행이 아니라, 순수한 위치 정보만 보여주면 충분하다.

### 3. Question Card

역할:

- 문제 본문을 메인 콘텐츠로 보여준다.

포함 대상:

- 문제 번호
- 문제 텍스트
- 필요 시 카테고리 표시

### 4. Choice List

역할:

- 선택 가능한 보기 목록의 구조를 먼저 만든다.

고려사항:

- 아직 선택 로직은 없지만, 이후 선택 강조가 들어갈 위치를 고려한 구조여야 한다.
- 보기는 목록으로 안정적으로 정렬되어야 한다.
- 보기 라벨은 이후 선택 브랜치에서 재사용 가능해야 한다.

### 5. Empty State

역할:

- 활성 문제 세트가 없을 때 사용자의 다음 행동을 안내한다.

포함 대상:

- 현재 문제 세트가 없다는 안내
- PDF 가져오기 또는 홈으로 이동 CTA

## Data Usage Notes

이번 단계에서는 활성 문제 세트의 실제 문제 데이터를 사용해야 한다.

- `loadQuestions()` 또는 동일한 데이터 접근 함수 사용
- 최소한 첫 번째 문제를 화면에 표시
- 문제 개수에 기반한 진행 정보 표시
- 활성 문제 세트 없음 상태를 별도 분기로 처리

이번 브랜치에서는 "모든 문제를 순회하는 로직"보다 "활성 문제 데이터가 정상적으로 화면에 연결된다"는 사실이 더 중요하다.

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- UI 단계에서 선택 로직까지 섞어 넣으면 다음 브랜치 경계가 흐려진다.
- 문제 데이터를 너무 깊게 가공하면 이후 로직과 책임이 겹칠 수 있다.
- 문제 카드와 보기 목록이 한 컴포넌트에 과도하게 뭉치면 유지보수가 어려워질 수 있다.
- 진행 상태를 실제 세션 로직처럼 과하게 설계하면 아직 하지 말아야 할 범위를 넘게 된다.
- 활성 문제 세트가 없는 경우를 무시하면 첫 실행 환경에서 빈 화면이나 오해를 만든다.

## Readability Revision

이후 수정에서는 문제 본문과 보기 텍스트의 가독성을 높이기 위해, 단순 문자열 출력이 아니라 읽기 흐름을 고려한 텍스트 렌더링 규칙을 추가해야 한다.

- 문제 본문은 긴 문장이어도 숨 막히게 붙어 보이지 않도록 줄간격과 단락 간격이 충분해야 한다.

- 문제 본문과 카테고리/문제 번호 사이의 정보 위계가 더 분명해야 한다.

## Range-aware Quiz Entry Revision

추가 수정에서는 퀴즈 화면이 더 이상 "활성 세트 전체를 바로 렌더링하는 화면"이 아니라, 홈에서 선택한 범위 정보를 받아 그 범위 안의 문제만 표시할 수 있어야 한다.

- 퀴즈 화면은 세트 전체 문제 수와 별도로 "이번 세션에 실제 포함된 범위"를 인식할 수 있어야 한다.
- 진행 표시는 세트 전체 기준이 아니라, 현재 범위 안 문제 수를 기준으로 계산되는 편이 자연스럽다.
- 헤더나 보조 정보에 필요하다면 현재 범위 문맥(`1~142`, `600~725` 등)을 짧게 보여줄 수 있어야 한다.
- 범위 필터 결과가 비어 있으면 기존의 "문제 세트 없음"과는 다른, "유효한 범위 안 문제 없음" 상태를 구분할 수 있어야 한다.
- 이 규칙은 일반/랜덤/시험 모드 공통의 화면 기반으로 쓰일 수 있어야 한다.

### Range-aware Quiz Entry Deliverables

- [x] 범위 필터 후 문제 목록을 기준으로 렌더링하는 퀴즈 화면 규칙
- [x] 범위 기준 진행 상태 표시
- [x] 필요 시 현재 범위 문맥 표시
- [x] 범위 결과 비어 있음 상태 처리

### Range-aware Quiz Entry Validation

- [x] 지정 범위 밖 문제는 퀴즈 화면에 나타나면 안 된다.
- [x] 진행 표시가 세트 전체가 아니라 범위 문제 수를 기준으로 맞아야 한다.
- [x] 잘못된 범위 또는 빈 범위는 이해 가능한 빈 상태로 이어져야 한다.

## Mobile Optimization Revision

추가 수정에서는 문제풀이 화면도 데스크톱 기준 여백과 카드 구조를 그대로 유지하지 않고, 모바일에서 한 문제를 한눈에 더 많이 읽을 수 있도록 밀도를 다시 조정해야 한다.

- 모바일에서는 헤더, 진행 정보, 문제 본문, 보기, 네비게이션이 가능한 한 한 화면 안에 더 많이 들어와야 한다.
- 문제 카드와 보기 카드의 불필요한 패딩, 큰 여백, 장식 요소는 모바일에서 더 줄어들 수 있어야 한다.
- 긴 문제 본문과 긴 보기 텍스트는 가독성을 유지하되, 카드 높이가 지나치게 커지지 않도록 균형을 맞춰야 한다.
- 해설, 결과 요약, 보조 문구는 모바일에서 기본적으로 덜 점유적인 구조가 되어야 한다.
- 네비게이션과 주요 CTA는 엄지로 누르기 쉬우면서도 화면을 과도하게 차지하지 않아야 한다.
- 시험/랜덤/일반 모드 공통으로 "한 문제를 보는 화면"의 밀도를 조정하는 방식이 적절하다.

이번 수정의 핵심은 문제풀이 화면을 "모바일에서도 바로 풀기 쉬운 밀도"로 재구성하는 것이다.

## Theme-aware Quiz UI Revision

추가 수정에서는 문제풀이 화면도 홈과 같은 전역 테마 규칙을 따라, 사용자가 `라이트 모드`와 `다크 모드` 사이를 전환해도 본질적인 읽기 흐름이 유지되어야 한다.

- 문제 화면은 전역 테마 설정을 읽어 배경, 문제 카드, 보기 카드, 진행 정보, 네비게이션 버튼 색을 함께 바꿀 수 있어야 한다.
- 다크 모드에서는 문제 본문과 선택지의 가독성을 유지하기 위해 카드와 페이지 배경 사이의 대비가 충분해야 한다.
- 라이트 모드에서는 현재처럼 밝은 화면을 유지하되, 세부 라벨과 배지류가 흐릿하게 사라지면 안 된다.
- 테마 전환은 홈에서만 보이는 설정이 아니라, 문제풀이와 결과/복습에도 일관되게 이어지는 편이 자연스럽다.
- 문제 화면은 테마가 바뀌어도 "문제 본문 우선"이라는 정보 위계를 잃지 않아야 한다.

### Theme-aware Quiz UI Deliverables

- [x] 전역 테마를 읽는 문제 화면 스타일 규칙
- [x] 문제 카드/보기 카드/네비게이션의 다크 모드 색 체계
- [x] 라이트 모드의 보조 라벨 및 배지 대비 재조정
- [x] 결과/복습 화면으로 확장 가능한 공통 테마 토큰 정리

### Theme-aware Quiz UI Validation

- [x] 라이트/다크 모드 전환 후 문제 본문 가독성이 유지되어야 한다.
- [x] 보기 선택, 정답/오답 하이라이트가 두 테마 모두에서 명확해야 한다.
- [x] 헤더, 진행 정보, 버튼이 테마 전환 후에도 정보 우선순위를 유지해야 한다.
- [x] 결과 화면과 복습 화면도 같은 테마 문맥 안에서 자연스럽게 보여야 한다.

## Dracula Dark Mode Fix Revision

추가 수정에서는 다크 모드에서 문제풀이 화면에 들어갔을 때 문제나 선택지가 배경에 묻혀 보이지 않는 문제를 해결하고, 전체 학습 흐름을 드라큘라 계열 다크 테마로 정리해야 한다.

- 문제풀이 화면은 다크 모드에서도 문제 본문, 보기, 진행 정보, 헤더가 모두 즉시 읽혀야 한다.
- 다크 모드 배경은 단순 검정이 아니라 드라큘라 계열의 짙은 남보라/남색 기반 표면 체계를 사용해야 한다.
- 문제 카드, 보기 카드, 결과 카드, 복습 완료 화면은 다크 배경 위에서 서로 분리되는 표면 명도 차를 가져야 한다.
- 전역 테마가 적용되어도 문제풀이 화면의 정답/오답 하이라이트, 버튼, 보조 라벨이 흐려지면 안 된다.
- 결과/복습/즐겨찾기/대시보드처럼 학습 흐름 안의 다른 화면도 같은 다크 모드 문맥을 유지하는 편이 자연스럽다.

### Dracula Dark Mode Fix Deliverables

- [x] 문제풀이 화면 가독성 이슈 수정
- [x] 드라큘라 계열 다크 모드 색상 체계 보강
- [x] 결과/복습/즐겨찾기/대시보드의 다크 모드 표면 정리
- [x] 다크 모드에서도 또렷한 버튼/라벨/상태 정보 유지

### Dracula Dark Mode Fix Validation

- [x] 결과/복습 화면도 지나치게 밝거나 검게 뜨지 않고 같은 테마 계열 안에서 보여야 한다.
- [x] 라이트/다크 전환 후에도 문제풀이 흐름이 끊기지 않아야 한다.
