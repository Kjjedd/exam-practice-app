# Codex 프로젝트 작업 지침서
## 시험 대비 예상문제풀이 웹앱 구축용

---

# 1. 문서 목적

이 문서는 Codex가 본 프로젝트를 일관되고 안정적으로 구현할 수 있도록 하기 위한 **프로젝트 전용 작업 지침서**이다.  
Codex는 본 문서를 프로젝트의 최상위 기준 문서로 삼아 작업해야 하며, 모든 구현·수정·리팩터링·버그 수정은 본 문서의 범위와 우선순위를 따라야 한다.

이 프로젝트는 **1인 개발**로 진행되며, **GitHub 브랜치 전략**과 **작업 단위 분리**를 엄격히 적용한다.

---

# 2. 프로젝트 개요

## 2.1 프로젝트명
시험 대비 예상문제풀이 웹앱

## 2.2 프로젝트 목표
핸드폰 및 컴퓨터 브라우저에서 모두 접근 가능한 반응형 웹앱을 구현한다.  
사용자는 객관식 문제를 풀고, 정답 여부를 즉시 확인하며, 해설을 보고, 결과를 요약해서 확인할 수 있어야 한다.  
또한 틀린 문제를 다시 복습하고, 랜덤 문제 출제, 시험 모드, 즐겨찾기, 학습 통계, 로컬 저장 기능까지 제공해야 한다.

## 2.3 개발 원칙
- 본 프로젝트는 개인용 학습 웹앱이다.
- 초기 개발은 MVP 중심으로 진행하되, 아래에 정의된 모든 기능을 필수 구현 대상으로 본다.
- 프로젝트 구조는 단순하지만 확장 가능하게 설계한다.
- PDF 원본 문제집은 앱 런타임에서 직접 소비하지 않고, 구조화된 문제 데이터(JSON 등)로 변환하여 사용하는 방향을 우선한다.
- 서버 없는 정적 웹앱 중심 구조를 우선으로 한다.
- 로컬 저장소를 활용하여 학습 이력을 유지한다.

# 4. 핵심 사용자 시나리오

## 4.1 기본 문제풀이 시나리오
1. 사용자가 웹앱에 접속한다.
2. 메인 화면에서 문제풀이를 시작한다.
3. 문제를 읽고 보기 중 하나를 선택한다.
4. 제출 버튼을 누른다.
5. 시스템이 정답 여부를 표시한다.
6. 사용자는 해설을 확인한다.
7. 다음 문제로 이동한다.
8. 모든 문제 풀이 후 결과 화면을 확인한다.

## 4.2 오답 복습 시나리오
1. 사용자가 결과 화면 또는 메인 화면에서 오답 문제 보기를 선택한다.
2. 틀린 문제 목록을 확인한다.
3. 각 문제의 정답과 해설을 다시 확인한다.
4. 틀린 문제만 다시 풀기를 진행한다.

# 6. 기능 상세 정의

## 6.1 객관식 문제 표시
- 문제 번호, 문제 본문, 보기 목록을 표시한다.
- 한 번에 한 문제 또는 문제 흐름 단위로 표시한다.
- 현재 문제 진행 상태를 알 수 있어야 한다.

## 6.2 보기 선택
- 사용자는 한 문제에 대해 하나의 보기를 선택할 수 있다.
- 선택한 보기는 시각적으로 강조되어야 한다.
- 제출 전에는 선택을 변경할 수 있어야 한다.

## 6.3 제출 및 정답 판정
- 제출 버튼을 누르면 정답 여부가 즉시 표시되어야 한다.
- 사용자의 선택 답안과 정답을 구분해서 보여줘야 한다.
- 같은 문제에 대한 중복 제출 정책은 명확히 처리해야 한다.
- 제출 전에는 채점 상태가 표시되면 안 된다.

## 6.4 해설 제공
- 제출 이후 해설을 볼 수 있어야 한다.
- 해설은 문제 이해에 도움이 되는 텍스트 형태로 표시한다.
- 필요 시 접기/펼치기 UI를 사용할 수 있다.

## 6.5 맞춘 문제 / 틀린 문제 분리 표시
- 결과 화면에서 정답 문제와 오답 문제를 구분해서 볼 수 있어야 한다.
- 필터링 또는 별도 화면 방식 모두 가능하다.
- 오답 문제만 다시 풀 수 있는 흐름이 제공되어야 한다.

## 6.6 결과 요약
- 총 문제 수
- 맞춘 문제 수
- 틀린 문제 수
- 정답률
- 필요 시 미응답 수
를 표시한다.

## 6.7 랜덤 문제 출제
- 사용자는 문제를 랜덤 순서로 풀 수 있어야 한다.
- 랜덤 모드 여부를 구분 가능해야 한다.
- 랜덤 출제 시에도 결과 계산은 정상적으로 동작해야 한다.

## 6.8 시험 모드
- 시험처럼 연속 풀이하는 모드를 제공한다.
- 시험 모드에서는 학습 모드와 다른 UI/UX를 적용할 수 있다.
- 최소한 결과 요약까지 자연스럽게 이어지는 흐름을 제공해야 한다.

## 6.9 문제 즐겨찾기
- 사용자는 특정 문제를 즐겨찾기할 수 있어야 한다.
- 즐겨찾기한 문제를 따로 모아볼 수 있어야 한다.

## 6.10 학습 통계 대시보드
- 정답률
- 누적 풀이 수
- 오답 비율
- 즐겨찾기 수
- 최근 학습 정보
등을 표시할 수 있어야 한다.

## 6.11 LocalStorage 기반 학습 기록 유지
- 학습 상태를 새로고침 후에도 유지한다.
- 정오답 기록, 즐겨찾기, 최근 학습 상태 등을 저장한다.
- 브라우저 저장소를 초기화할 수 있는 옵션을 고려한다.

# 7. 비기능 요구사항

## 7.1 반응형 웹
- 모바일, 태블릿, 데스크톱 화면 크기에 대응해야 한다.
- 터치 환경에서도 문제풀이가 불편하지 않아야 한다.

## 7.2 유지보수성
- 컴포넌트와 로직은 명확히 분리한다.
- 타입 정의를 일관성 있게 관리한다.
- 문제 데이터 구조를 고정하고 재사용 가능하게 유지한다.

## 7.3 확장성
- 추후 PDF → JSON 자동 변환 스크립트 연결 가능성을 고려한다.
- 향후 IndexedDB, 백엔드, 인증, 동기화로 확장 가능한 구조를 유지한다.

## 7.4 가독성
- 지나치게 복잡한 추상화는 피한다.
- 파일명, 함수명, 상태명은 목적이 드러나게 작성한다.
- 생성 코드에는 필요한 수준의 명확한 주석을 포함할 수 있다.

# 8. 데이터 구조 원칙

## 8.1 문제 데이터 예시
```json
{
  "id": "q-001",
  "question": "OSI 7계층 중 전송 계층은 무엇인가?",
  "choices": [
    "물리 계층",
    "전송 계층",
    "응용 계층",
    "세션 계층"
  ],
  "answer": 1,
  "explanation": "전송 계층은 종단 간 통신을 담당한다.",
  "category": "network"
}
```

## 8.2 풀이 기록 예시
```json
{
  "questionId": "q-001",
  "selectedAnswer": 2,
  "isCorrect": false,
  "isFavorite": false,
  "solvedAt": "2026-04-10T14:00:00"
}
```

## 8.3 주의사항
- answer는 choices의 인덱스 또는 명확한 정답 식별 방식으로 일관되게 정의한다.
- 데이터 구조는 컴포넌트와 로직 전반에서 동일하게 유지한다.
- Codex는 데이터 구조를 임의로 자주 변경하지 말고, 필요 시 변경 이유를 먼저 설명해야 한다.

# 10. GitHub 브랜치 전략

본 프로젝트는 1인 개발이므로, 무거운 Git Flow 대신 **가벼운 feature branch 전략**을 사용한다.

## 10.1 기본 브랜치 정책
- `main`: 항상 비교적 안정적인 상태 유지
- 직접 `main`에서 작업하지 않는다.
- 모든 작업은 별도 브랜치에서 수행 후 병합한다.

## 10.2 브랜치 종류
- `feature/*`: 기능 개발
- `fix/*`: 버그 수정
- `refactor/*`: 리팩터링
- `chore/*`: 설정/정리 작업
- `experiment/*`: 실험용 작업

## 10.3 브랜치 이름 예시
- `feature/question-ui`
- `feature/answer-checking`
- `feature/explanation-panel`
- `feature/wrong-note`
- `feature/random-quiz-mode`
- `feature/exam-mode`
- `feature/favorites`
- `feature/statistics-dashboard`
- `feature/localstorage-progress`
- `fix/result-summary-bug`

## 10.4 브랜치 운영 원칙
- 브랜치 하나는 작업 하나를 담당한다.
- 하나의 브랜치에서 여러 기능을 동시에 구현하지 않는다.
- 작업 완료 후 self-review 또는 PR 검토 후 병합한다.
- 병합 후 불필요한 브랜치는 삭제한다.

# 11. 커밋 전략

## 11.1 기본 원칙
- 작은 단위로 자주 커밋한다.
- 하나의 커밋은 하나의 의도를 가져야 한다.
- 커밋 메시지는 목적이 분명해야 한다.

## 11.2 권장 커밋 타입
- `feat:`
- `fix:`
- `refactor:`
- `chore:`
- `docs:`
- `test:`

## 11.3 커밋 메시지 예시
- `feat: add question solving page`
- `feat: implement answer checking logic`
- `feat: add explanation toggle UI`
- `feat: implement wrong answer review flow`
- `feat: add random question mode`
- `feat: add exam mode flow`
- `feat: add favorites feature`
- `feat: add statistics dashboard`
- `feat: persist quiz progress to localStorage`
- `fix: correct result summary calculation`

## 11.4 병합 전략
- feature 브랜치 내부에서는 자유롭게 커밋 가능
- `main` 병합 시에는 히스토리가 지저분하지 않도록 정리
- squash merge를 우선 고려

# 12. Codex 작업 원칙

Codex는 아래 원칙을 반드시 따른다.

## 12.1 계획 우선
코드 생성 전에 반드시 아래 순서를 따른다.
1. 작업 목표 이해
2. 구현 계획 제시
3. 수정/생성할 파일 목록 제시
4. 실제 코드 작업 수행

## 12.2 작업 범위 제한
- 현재 브랜치의 목적과 무관한 변경은 하지 않는다.
- 요청되지 않은 대규모 구조 변경은 하지 않는다.
- 여러 기능을 한 번에 섞어서 구현하지 않는다.

## 12.3 단계별 구현
Codex는 한 번에 전체 프로젝트를 완성하려 하지 말고, 아래처럼 기능 단위로 나눠 진행한다.
1. 프로젝트 초기화
2. 타입 정의
3. 샘플 데이터 연결
4. 문제풀이 UI
5. 정답 판정
6. 해설
7. 결과 요약
8. 오답 복습
9. 랜덤 모드
10. 시험 모드
11. 즐겨찾기
12. 통계 대시보드
13. LocalStorage 저장

## 12.4 검증 우선
각 작업 후 다음을 고려한다.
- 타입 오류 여부
- 빌드 가능 여부
- 화면 흐름 정상 여부
- 상태 관리 일관성 여부

## 12.5 설명 의무
다음 상황에서는 변경 이유를 설명해야 한다.
- 폴더 구조 변경
- 데이터 구조 변경
- 상태 구조 변경
- 주요 컴포넌트 분리/통합
- 저장 방식 변경

# 13. Codex에게 금지되는 작업 방식

Codex는 아래와 같은 작업을 피해야 한다.

- 요구사항에 없는 임의 기능 추가
- 브랜치 목적과 무관한 파일 수정
- 설명 없는 구조 전면 개편
- 여러 핵심 기능을 한 번에 섞은 대규모 수정
- 임시 코드와 실제 코드를 섞어 배치
- 지나치게 복잡한 추상화
- 검증 없이 저장 로직을 깊게 얽는 구조
- 사용자가 요청하지 않은 백엔드/DB 도입

# 14. 우선 구현 순서

Codex는 아래 순서대로 구현을 제안해야 한다.

## 14.1 1차 구현
- 프로젝트 초기화
- 기본 폴더 구조
- 타입 정의
- 샘플 문제 데이터
- 메인 화면
- 문제풀이 화면

## 14.2 2차 구현
- 보기 선택
- 제출
- 정답 판정
- 해설 제공
- 다음 문제 이동
- 결과 화면

## 14.3 3차 구현

# 17. 최종 지침

Codex는 본 문서를 프로젝트 전용 개발 규칙으로 간주해야 한다.  
모든 작업은 다음 원칙을 따른다.

- 요구사항 준수
- 브랜치 단위 작업
- 단계별 구현
- 사람 검토 가능성 확보
- 유지보수 가능한 코드 작성
- 설명 가능한 변경만 수행

본 문서와 충돌하는 방향의 구현을 제안하거나 수행해서는 안 된다.
