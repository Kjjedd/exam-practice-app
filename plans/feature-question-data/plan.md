# feature/question-data Plan

## Goal

검수 완료된 문제 세트를 저장하고, 이후 브랜치가 활성 문제 세트를 일관되게 읽을 수 있는 데이터 접근 구조를 만든다.

## Branch Intent

이 브랜치는 "검수 완료된 문제 세트를 영구 저장하고, 앱이 소비할 활성 문제 데이터로 노출한다"는 목적만 담당한다.

- [x] 검수 완료된 문제 세트를 저장 가능한 구조로 정리한다.
- [x] 현재 활성 문제 세트를 읽는 접근 방식을 단순하고 재사용 가능하게 만든다.
- [x] 이후 문제 UI와 채점 로직이 저장된 문제 세트를 바로 사용할 수 있게 한다.

이 단계에서는 저장된 문제 세트와 데이터 접근 구조만 다루고, 문제풀이 화면 동작은 구현하지 않는다.  
기존의 정적 샘플 JSON 중심 구현은 초기 검증용 결과였으며, 현재 계획에서는 PDF 검수 후 저장 구조로 대체된다.

## Scope

- 검수 완료된 문제 세트 저장 구조 정의
- 활성 문제 세트 메타데이터 정의
- 저장된 문제 세트 읽기/쓰기 인터페이스 정리
- 문제 ID, 보기, 정답, 해설 구조 일관성 확보
- 이후 브랜치에서 재사용 가능한 데이터 로더 유틸 작성

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 검수 완료된 문제 세트를 브라우저에 안전하게 저장하는 방식 정리
- 활성 문제 세트가 없을 때의 기본 동작 정의
- 다음 브랜치가 문제 목록 전체와 단일 문제 조회를 쉽게 사용할 수 있는 접근 함수 준비
- 저장된 데이터가 타입과 실제 사용 흐름을 동시에 만족하는지 검증

이번 브랜치에서는 "저장된 문제 세트의 신뢰성"이 핵심이며, UI 편의를 위한 화면 구현은 넣지 않는다.

## Data Principles

저장되는 문제 세트는 아래 원칙을 따라야 한다.

- 모든 문제는 고유한 `id`를 가진다.
- 모든 문제는 `Question` 타입과 일치한다.
- `answer`는 반드시 `choices` 인덱스 범위 안에 있어야 한다.
- 저장이 끝난 세트는 이후 문제풀이에서 추가 정규화 없이 사용할 수 있어야 한다.
- 활성 문제 세트와 과거 저장 세트의 경계를 구분할 수 있어야 한다.
- 문제 데이터는 이후 랜덤 모드, 결과 집계, 오답 복습까지 견딜 수 있는 기본 품질을 가져야 한다.

## Proposed Data Shape

이번 단계에서는 이미 정의된 `Question` 타입을 기준으로, 저장 단위는 아래 두 층으로 나뉘는 것이 적절하다.

1. 단일 문제 레코드
2. 문제 세트 레코드

문제 세트 레코드는 최소한 아래 정보를 가져야 한다.

- 세트 `id`
- 세트 이름 또는 업로드 출처 메타데이터
- 생성 시각
- 문제 배열
- 현재 활성 세트 여부 또는 활성 세트를 가리킬 수 있는 참조 정보

지금 목표는 저장 단위를 만들되, 이후 문제풀이가 불필요한 래핑 없이 `Question[]`를 읽을 수 있는 구조를 만드는 것이다.

## Loader Strategy

데이터 접근 유틸은 최대한 단순해야 한다.

가능한 책임:

- 현재 활성 문제 세트 조회
- 활성 세트의 전체 문제 목록 반환
- 문제 ID 기반 단일 문제 조회 또는 조회에 필요한 기초 제공
- 검수 완료 데이터 저장
- 이후 브랜치에서 재사용 가능한 import 경로 정리

이번 단계에서 피해야 하는 것:

- 과도한 캐싱 구조
- 필터링/셔플 같은 비즈니스 로직
- 저장소와 세션 상태를 한 함수에 뒤섞는 처리

## Out of Scope

- 문제풀이 UI
- 채점 로직
- 결과 계산 로직
- 업로드/검수 화면 구현
- 통계 기능 구현
- 즐겨찾기 기능 구현
- 문제 선택 상태
- 결과 타입 변경

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- 문제 데이터에 사용자 세션 상태까지 섞어 넣으면 책임이 흐려진다.
- 로더 유틸이 비즈니스 로직까지 맡으면 다음 브랜치에서 다시 분리가 필요해질 수 있다.

- 활성 세트 개념을 느슨하게 만들면 홈/퀴즈/결과 흐름이 어느 데이터를 기준으로 동작하는지 모호해진다.

## Range-limited Practice Revision

첫 번째 신규 요구사항은 세트를 더 잘게 쪼개는 방향이 아니라, 현재 활성화된 기본 세트 안에서 사용자가 직접 문제 범위를 지정해 푸는 구조로 바뀌었다.

- 범위 지정 대상은 우선 현재 활성화 가능한 두 기본 세트다.
- `aws-saa-default`
- `aws-saa-600-plus`
- 각 세트는 홈 또는 시작 화면에서 사용자에게 "이 세트에서 지정 가능한 최소/최대 번호"를 안내할 수 있어야 한다.
- 예를 들어 `aws-saa-default`는 `1~725`, `aws-saa-600-plus`는 해당 세트의 내부 번호 범위를 기준으로 사용자가 시작 번호와 끝 번호를 선택할 수 있어야 한다.
- 저장소와 데이터 계층은 세트의 실제 문제 배열 외에도 "사용자에게 보여줄 번호 범위"를 계산할 수 있어야 한다.
- 범위는 문제 배열 인덱스가 아니라, 사용자가 이해하는 문제 번호 기준으로 다뤄야 한다.
- 지정 범위는 시작값과 끝값을 모두 포함하는 구간으로 해석하는 편이 자연스럽다.
- 잘못된 범위(`start > end`, 세트 최소/최대 범위를 벗어남)는 데이터 계층에서 안전하게 검증할 수 있어야 한다.
- 이후 일반/랜덤/시험 모드는 모두 이 범위 필터를 거친 문제 집합을 기준으로 세션을 만들어야 한다.

이번 수정의 핵심은 "기본 세트 수를 늘리는 것"이 아니라, 하나의 활성 세트를 다양한 학습 범위로 재사용할 수 있게 하는 데이터 기준을 마련하는 것이다.

- [x] 세트와 범위를 입력받아 유효한 문제 부분집합을 반환하는 데이터 계층 기준

## Built-in Set Cleanup Revision

추가 수정에서는 기본 문제 은행을 다시 단순화해서, 현재 앱에 노출되는 기본 세트는 아래 두 개만 남겨야 한다.

- `aws-saa-default`
- `aws-saa-600-plus`

기존에 임시로 추가했던 아래 기본 세트는 더 이상 홈 화면과 저장소에 남아 있으면 안 된다.

- `aws-saa-1-100`
- `aws-saa-600-725`

이번 수정의 핵심은 단순히 import를 지우는 것이 아니라, 이미 브라우저 저장소에 남아 있는 예전 기본 세트도 기본 병합 과정에서 자연스럽게 제거되도록 만드는 것이다.

- 기본 문제 은행은 위 두 세트만 공식 내장 세트로 관리해야 한다.
- 저장소 병합 시 사용자 업로드 세트는 유지하되, 제거 대상 예전 기본 세트는 다시 나타나면 안 된다.
- 활성 세트가 제거 대상 구세트를 가리키고 있었다면, 유효한 현재 기본 세트 중 하나로 안전하게 되돌려야 한다.
- 현재 유지하는 두 세트의 제목도 더 읽기 쉬운 라벨로 정리하는 편이 좋다.

## Canonical AWS SAA Merge Revision

이번 추가 요구사항의 핵심은 현재 분리되어 있는 두 개의 기본 AWS SAA 세트를 단순히 합치는 것이 아니라, **번호가 겹치지 않고 정답 신뢰도를 최대한 확보한 단일 통합 세트**를 만드는 것이다.

대상 세트는 아래 두 개다.

- 기준 세트: `aws-saa-default`
- 보강 세트: `aws-saa-600-plus`

현재 전제는 다음과 같다.

- `aws-saa-default`는 1~725를 대표하는 기준 세트다.
- 다만 이 세트에는 실제로 비어 있는 번호가 존재한다.
- `aws-saa-600-plus`는 600~1019를 다루지만 정답이 잘못 기입된 문제가 다수 있을 수 있다.
- 두 세트는 600~725 구간에서 번호 기준으로 겹친다.
- 따라서 `aws-saa-600-plus`의 726~1019를 그대로 믿고 붙이는 것은 위험하다.

이번 리비전의 목표는 아래 순서를 엄격히 따르는 것이다.

1. `aws-saa-default`의 1~725 구간을 먼저 가능한 한 완전체로 복구한다.
2. 복구에 쓰이는 `aws-saa-600-plus` 구간의 정답 품질을 600~725 겹침 영역에서 먼저 검증한다.
3. 그 뒤에만 726~1019를 통합 세트에 이식한다.
4. 최종적으로 단일 AWS SAA 통합 세트를 만든다.

### Revision Intent

이번 수정은 단순 데이터 추가 작업이 아니라, **기준 세트 우선 원칙 + 보강 세트 검증 기반 병합**이 핵심이다.

- 1~725 구간에서는 `aws-saa-default`가 기준 진실(source of truth)이다.
- `aws-saa-default`에서 비어 있는 번호만 `aws-saa-600-plus`에서 보강 대상으로 삼는다.
- `aws-saa-600-plus`의 정답은 자동으로 신뢰하지 않는다.
- 600~725 겹침 구간은 `aws-saa-600-plus`의 품질을 평가하는 검증 샘플 구간으로 사용한다.
- 726~1019는 검증 규칙을 통과한 뒤에만 단일 통합 세트로 이식한다.

### Phase 1: 1~725 기준 세트 실측

첫 단계는 `aws-saa-default`를 실제 번호 기준으로 분석하는 것이다.

필요한 작업:

- `aws-saa-default`의 실제 번호 목록 추출
- 1~725 범위에서 존재 번호와 누락 번호 분리
- 누락 번호를 정렬된 목록으로 관리
- 누락 번호가 `aws-saa-600-plus`에 존재하는지 확인

이번 단계에서 중요한 점은, "비어 있는 번호"와 "나중에 보강 가능한 번호"를 분리해서 다뤄야 한다는 것이다.

예상 산출물:

- `defaultPresentNumbers`
- `defaultMissingNumbersWithin1To725`
- `candidateBackfillNumbers`

### Phase 2: 600~725 겹침 구간 감사(Audit)

둘째 단계는 겹치는 번호를 이용해 `aws-saa-600-plus`의 품질을 검증하는 것이다.

이 구간은 사실상 보강 세트의 기준 샘플이다.  
왜냐하면 726 이후 번호는 기준 세트와 직접 비교할 수 없기 때문이다.

겹침 구간 비교 기준:

1. 번호가 같은가
2. 문제 본문 의미가 같은가
3. 보기 구조가 같은가
4. 정답이 같은가
5. 해설 또는 추가 블록(정책/코드/JSON)이 논리적으로 일치하는가

주의할 점:

- 단순 문자열 완전 일치에만 의존하면 안 된다.
- OCR 차이, 공백, 개행, 코드 블록 형식 차이가 있을 수 있다.
- 따라서 정규화 비교가 필요하다.

정규화 예시:

- 연속 공백 압축
- 불필요한 줄바꿈 제거
- 코드블록 래퍼(````json`) 통일
- 큰따옴표/유니코드 따옴표 차이 무시

번호별 결과는 최소 아래 상태로 분류하는 것이 적절하다.

- `match_exact`
- `match_text_variant_same_answer`
- `match_text_variant_different_answer`
- `question_mismatch_same_number`
- `missing_in_default`
- `missing_in_600_plus`

이 단계의 목적은 아래 질문에 답하는 것이다.

- `aws-saa-600-plus`는 겹침 구간에서 얼마나 자주 기준 세트와 같은 정답을 제공하는가?
- 답이 다를 때는 OCR 차이인지 실제 오답 데이터인지 설명 가능한가?
- 특정 유형의 문항에서 오답 경향이 집중되는가?

필요 산출물:

- `overlapAuditReport`
- `overlapVerifiedMatches`
- `overlapAnswerConflicts`
- `overlapQuestionConflicts`
- `overlapReliabilitySummary`

### Phase 3: 1~725 완전체 복구

셋째 단계는 `aws-saa-default`의 빈 번호만 조심스럽게 채워 넣는 것이다.

적용 규칙:

- 기준 세트에 이미 존재하는 번호는 그대로 유지
- 기준 세트에 없는 번호만 보강 세트에서 가져옴
- 단, 해당 번호가 보강 후보이고 품질 규칙상 이식 가능해야 함
- 충돌이 있는 번호는 자동 이식하지 않음

이번 단계의 결과는 "1~725 기준 완전체"다.

목표:

- 기준 세트의 정체성 유지
- 비어 있는 번호 최소화
- 이후 726~1019 확장에 앞서 안정적인 전반부 세트 확보

필요 산출물:

- `canonicalBaseQuestionSet1To725`
- `backfilledQuestionNumbers`
- `unresolvedMissingQuestionNumbers`

### Phase 4: 726~1019 이식

넷째 단계는 726 이후를 최종 통합 세트에 추가하는 것이다.

이 구간은 기준 세트가 없으므로, 앞서 600~725에서 얻은 신뢰도 결과를 기반으로 조건부 이식해야 한다.

적용 규칙:

- 726~1019는 번호 충돌이 없으므로 기본적으로 이식 대상이다.
- 하지만 아래 문제는 자동 이식 전에 경고 또는 보류 대상으로 분류하는 편이 좋다.

검토 대상 예시:

- 보기 개수가 비정상적인 문제
- `answers` 인덱스가 `choices` 범위를 벗어나는 문제
- 문제 본문이 지나치게 짧거나 잘린 문제
- 정책/코드/JSON 블록이 깨진 문제
- 다중 정답처럼 보이는데 정답 수가 단일인 문제
- 해설 논리가 문제/보기와 맞지 않는 문제

필요 산출물:

- `tailCandidates726To1019`
- `tailValidationReport`
- `acceptedTailQuestions`
- `flaggedTailQuestionsForManualReview`

### Phase 5: 최종 단일 세트 생성

다섯째 단계는 위 결과를 사용해 최종 통합 세트를 생성하는 것이다.

최종 세트 규칙:

- 번호는 오름차순
- 같은 번호는 한 번만 존재
- `QuestionSet` 타입과 완전 일치
- 앱에서 곧바로 활성 세트로 사용할 수 있어야 함

운영상 더 안전한 절차는 아래와 같다.

1. 기존 `aws-saa-default`를 바로 덮어쓰지 않는다.
2. 먼저 새 통합 세트(예: `aws-saa-canonical`)를 만든다.
3. 검증이 끝나면 기본 활성 세트 교체 여부를 결정한다.

### Source-of-Truth Rules

이번 작업에서 번호별 출처 우선순위는 명확해야 한다.

- 1~725
  - 1순위: `aws-saa-default`
  - 보강: `aws-saa-600-plus` (기준 세트에 번호가 없을 때만)
- 726~1019
  - 유일 출처: `aws-saa-600-plus`
  - 단, 겹침 구간 감사 결과를 기반으로 신뢰 조건을 통과해야 함

즉 "두 세트의 겹치는 문제를 모두 남긴다"는 선택지는 없다.  
반드시 단일 번호, 단일 문항 구조로 정리해야 한다.

- 어떤 번호는 왜 자동 이식되지 않았는가?

### Risks for This Revision

이번 리비전의 주요 위험은 아래와 같다.

- 번호는 같지만 실제 다른 문제일 수 있다.
- OCR 차이로 본문이 달라도 사실상 같은 문제일 수 있다.
- 겹침 구간에서 답이 다를 때 자동 판정이 어려울 수 있다.
- 726~1019는 기준 세트가 없어 완전 자동 검증이 제한적이다.
- 잘못된 자동 병합은 이후 오답복습/즐겨찾기/시험모드 전체를 오염시킬 수 있다.

따라서 이번 작업은 "빠른 병합"보다 "설명 가능한 병합"이 더 중요하다.
