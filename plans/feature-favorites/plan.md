# feature/favorites Plan

## Goal

사용자가 문제를 즐겨찾기하고, 별도 즐겨찾기 화면에서 저장된 문제 목록을 확인한 뒤 다시 문제풀이 흐름으로 진입할 수 있게 한다.

## Branch Intent

이 브랜치는 "세션 단위 학습" 위에 "문제 단위 개인 보관함"을 추가하는 단계다.

- [x] 현재 문제를 즐겨찾기/해제할 수 있는 토글 흐름을 만든다.
- [x] 즐겨찾기한 문제를 별도 화면에서 목록으로 확인할 수 있게 한다.
- [x] 즐겨찾기 목록에서 다시 문제풀이 흐름으로 진입할 수 있게 한다.
- [x] 다음 단계인 저장/진행 관리 기능과 충돌하지 않도록 즐겨찾기 책임을 분명히 유지한다.

이번 단계의 핵심은 "문제를 개인적으로 저장해 두고 다시 찾아볼 수 있게 하는 것"이지, 장기 학습 통계를 계산하는 것이 아니다.

## Scope

- 문제 화면 즐겨찾기 토글
- 즐겨찾기 상태 저장/복원
- 즐겨찾기 목록 화면
- 즐겨찾기 문제 재진입 흐름
- 즐겨찾기 문제가 없을 때의 빈 상태 처리

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 현재 문제를 `questionId` 기준으로 즐겨찾기/해제할 수 있게 하기
- 즐겨찾기 상태를 브라우저 저장소에 보존해 새로고침 후에도 유지되게 하기
- `/favorites` 화면에서 저장된 문제 목록을 안정적으로 렌더링하기
- 목록에서 특정 문제로 다시 이동할 수 있는 진입 경로를 만들기
- 활성 문제 세트가 바뀌어도 저장된 즐겨찾기 목록을 안전하게 읽을 수 있게 하기

이번 브랜치에서는 즐겨찾기를 세션 통계나 오답 복습과 결합하지 않는다. 즐겨찾기는 독립적인 개인 보관 기능으로 다룬다.

## State Principles

즐겨찾기 상태는 아래 원칙을 따라야 한다.

- 즐겨찾기 식별 기준은 `questionId`여야 한다.
- 문제 세트 순서나 모드와 무관하게 같은 문제는 같은 즐겨찾기 항목으로 취급해야 한다.
- 저장된 즐겨찾기와 현재 활성 문제 세트의 문제 목록은 분리해서 생각해야 한다.
- 즐겨찾기 목록에 없는 문제는 토글 시 새로 추가되고, 이미 있으면 제거되어야 한다.
- 즐겨찾기 상태는 결과 집계나 복습 흐름과 직접 섞이면 안 된다.

## Proposed Data Shape

이번 단계에서는 최소한 아래 수준의 구조가 필요할 가능성이 높다.

- `favoriteQuestionIds: QuestionId[]`
- 필요 시 즐겨찾기 목록 화면용 유도 값:
  - `favoriteQuestions: Question[]`
  - `favoriteCount: number`

핵심은 저장 포맷은 단순하게 두고, 실제 문제 데이터는 렌더링 시 활성 문제 세트와 연결해 해석하는 것이다.

## Favorite Rules

즐겨찾기 규칙은 최소한 아래를 만족해야 한다.

- 토글은 같은 문제를 중복 저장하지 않아야 한다.
- 해제는 정확히 해당 `questionId`만 제거해야 한다.
- 목록 순서는 저장 순서 또는 현재 문제 세트 순서 중 하나로 일관되게 유지해야 한다.
- 즐겨찾기 목록은 현재 활성 문제 세트에서 찾을 수 없는 문제를 안전하게 처리해야 한다.

## Interaction Design

이번 단계에서 기대하는 상호작용은 다음과 같다.

### 1. In Quiz

- 사용자는 현재 보고 있는 문제를 즐겨찾기할 수 있다.
- 이미 즐겨찾기된 문제라면 해제할 수 있다.
- 상태 변화는 화면에서 즉시 확인 가능해야 한다.

### 2. In Favorites Page

- 사용자는 저장된 즐겨찾기 문제 목록을 본다.
- 각 문제의 핵심 정보와 다시 풀기 진입점을 확인할 수 있다.

### 3. Re-entry

- 사용자가 즐겨찾기 목록에서 문제를 선택하면 해당 문제로 다시 진입할 수 있어야 한다.
- 이 진입은 기존 퀴즈 흐름과 너무 동떨어지면 안 된다.

### 4. Empty State

- 즐겨찾기한 문제가 없으면 빈 상태 메시지와 홈/퀴즈로 돌아가는 안내가 필요하다.

### 5. Non-goals in This Step

- 즐겨찾기 폴더 분류
- 태그 추가
- 즐겨찾기 정렬/검색
- 즐겨찾기 통계 반영

## UI Principles

- 즐겨찾기 토글은 문제 화면에서 눈에 띄되, 문제 읽기 흐름을 방해하면 안 된다.
- 목록 화면은 저장된 문제를 빠르게 훑어볼 수 있어야 한다.
- 빈 상태는 사용자가 왜 목록이 비었는지 쉽게 이해할 수 있어야 한다.
- 즐겨찾기 화면은 결과/복습 화면과 역할이 다르므로, "개인 보관함" 느낌이 분명해야 한다.

## Data Flow Notes

현재 프로젝트는 활성 문제 세트와 최신 세션 결과를 저장하고 있다.  
이번 단계에서는 아래 흐름이 중요하다.

- 문제 화면에서 `questionId` 기준 즐겨찾기 토글
- 브라우저 저장소에 즐겨찾기 ID 목록 저장
- `/favorites`가 저장된 ID 목록을 읽고 현재 문제 데이터와 연결
- 다시 진입 시 해당 문제를 퀴즈 화면에서 찾을 수 있도록 연결

즉, 즐겨찾기의 핵심은 "문제 ID 보관"이지, 전체 문제 데이터를 중복 저장하는 것이 아니다.

## Risks

- 즐겨찾기 저장과 문제 데이터 저장을 한 구조에 섞으면 책임이 흐려질 수 있다.
- 활성 문제 세트가 바뀐 상황에서 없는 문제를 잘못 렌더링하면 목록 화면이 깨질 수 있다.
- 재진입 흐름이 퀴즈 화면 구조와 과도하게 결합되면 유지보수가 어려워질 수 있다.
- 모드별 분기가 즐겨찾기 로직까지 침투하면 문제 단위 기능이라는 장점이 사라질 수 있다.

- 단순한 저장 구조와 명확한 재진입 경험이 가장 중요하다.
