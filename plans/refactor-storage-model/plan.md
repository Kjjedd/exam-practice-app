# refactor/storage-model Plan

## Goal

현재 동작 중인 저장 계층을 더 명확한 책임 단위로 정리해서, 이후 PDF 가져오기 흐름과 추가 기능이 들어와도 저장 구조가 흔들리지 않도록 유지보수성을 높인다.

## Branch Intent

이 브랜치는 사용자 기능을 새로 추가하는 단계가 아니라, 이미 만들어진 저장 계층을 "더 설명 가능하고 더 안전한 구조"로 다듬는 단계다.

- [x] 저장 책임이 중복되거나 모호한 부분을 정리한다.
- [x] 데이터 계층, 저장 계층, 화면 계층이 서로의 내부 구현을 덜 알도록 경계를 개선한다.
- [x] 검증, 읽기/쓰기, 요약 계산, 초기화 로직이 어디에 있어야 하는지 기준을 분명히 한다.
- [x] 이후 PDF 업로드/검수 흐름이 저장 구조를 확장하더라도 큰 재작업이 없도록 기반을 정리한다.

이번 단계의 핵심은 "파일 몇 개 옮기기"가 아니라, 현재 구조에서 생길 수 있는 장기 유지보수 비용을 미리 낮추는 것이다.

## Scope

- 저장 책임 경계 재정리
- 세션 데이터와 문제 세트 데이터 경계 재검토
- 즐겨찾기/세션/문제 세트 저장 API 표면 정리
- 저장 키/초기화/검증 책임 구조 검토
- 대시보드 같은 해석 계층이 저장 구현에 과하게 의존하지 않도록 구조 개선

## Detailed Scope

이번 단계에서 검토하고 정리해야 할 핵심은 아래와 같다.

- `lib/data/save-question-set.ts`와 `lib/storage/question-bank.ts` 사이의 책임 분리를 더 명확히 하기
- `question bank`, `latest quiz session`, `favorites`의 읽기/쓰기/초기화 규칙을 가능한 한 일관된 형태로 정리하기
- `session-storage` 안에 있는 세션 검증, 파싱, 완결성 판단이 다른 계층에서 재사용되는 방식 점검하기
- `dashboard`나 `result` 같은 해석 계층이 저장 구현 세부사항에 직접 의존하는 부분이 있는지 검토하기
- 저장 관련 공통 유틸이 `storage`에 있어야 할지, `data` 또는 `dashboard`에 남아야 할지 경계를 다시 점검하기
- PDF 기반 문제 세트가 본격적으로 들어왔을 때도 현재 저장 모델이 버틸 수 있는지 확인하기

이번 브랜치에서는 사용자에게 보이는 기능을 늘리기보다, 이미 동작하는 구조의 내부 품질을 높이는 데 집중한다.

## Current Structural Observations

현재 저장 계층을 보면 다음과 같은 특징이 있다.

### 1. Strengths

- 문제 세트, 최신 세션, 즐겨찾기가 개별 저장소로 이미 분리돼 있다.
- 기본적인 파싱 실패 복구와 기본값 반환이 들어가 있다.
- 전체 저장 상태 요약 유틸과 전체 초기화 유틸이 생겨 저장소 관점이 조금씩 통합되고 있다.

### 2. Refactor Signals

- 문제 세트 관련 로직이 `lib/data`와 `lib/storage`에 나뉘어 있어 경계가 다소 애매하다.
- 검증 함수와 저장 함수가 같은 파일 안에 몰려 있어, 역할별 분리가 더 가능하다.
- 세션 완결성 판단과 대시보드 계산 같은 "해석 로직"이 저장소 인터페이스와 가까워지기 시작했다.
- 저장소별 API 모양이 조금씩 달라서, 장기적으로는 일관된 사용 규칙이 필요하다.

즉, 지금 구조는 충분히 동작하지만, 앞으로 PDF 가져오기/검수 기능이 본격적으로 연결되면 복잡도가 빠르게 올라갈 수 있는 상태다.

## Refactor Principles

리팩터링은 아래 원칙을 따라야 한다.

- 동작 중인 사용자 기능을 바꾸지 않는다.
- 저장 포맷을 바꾸더라도 호환성 전략이 없으면 조심스럽게 접근한다.
- 구조를 더 설명 가능하게 만들되, 과도한 추상화는 피한다.
- 읽기/쓰기/검증/초기화/요약 해석은 서로 다른 책임으로 본다.
- 파일 수를 늘리는 것이 목적이 아니라, 책임을 읽기 쉽게 만드는 것이 목적이다.
- 리팩터 후에는 각 계층이 "무엇을 알고 무엇을 몰라야 하는지"가 더 명확해야 한다.

## Proposed Refactor Areas

### 1. Storage Core

- 저장 키 정의
- 로컬 저장소 접근 가능 여부 확인
- 기본 파싱/기본값 복구 패턴 정리

이 부분은 여러 저장소에서 반복되는 구조가 있으므로, 공통화 가능성을 검토할 수 있다. 다만 과도하게 일반화하지는 않는다.

### 2. Question Bank Boundary

- 문제 세트 검증
- 문제 세트 저장/활성화
- 문제 세트 요약

현재는 `data`와 `storage`가 함께 관여하므로, 어느 계층이 "저장된 문제 세트 관리"를 대표하는지 더 분명히 할 필요가 있다.

### 3. Session Boundary

- 세션 읽기/쓰기
- 세션 완결성 판단
- 세션 파싱/검증

세션 완료 여부 판단은 여러 화면에서 쓰이므로, 저장 유틸에 둘지 별도 도메인 유틸로 분리할지 검토 대상이다.

### 4. Favorites Boundary

- ID 목록 저장
- 토글/초기화
- 렌더링 해석은 화면 또는 별도 해석 유틸에서 처리

즐겨찾기는 현재 단순해 보이지만, 문제 세트가 바뀌는 상황을 고려하면 "보관"과 "해석"을 계속 분리하는 게 중요하다.

### 5. Derived Read Models

- 대시보드 요약
- 결과 요약
- 오답 복습용 파생 데이터

이 부분은 저장소 자체가 아니라 "읽기 모델(read model)"에 가깝기 때문에, 저장 계층과 너무 섞이지 않도록 경계를 점검해야 한다.

## Proposed Structural Targets

리팩터 후 목표 구조는 대략 아래와 같다.

- `storage`
  - 실제 브라우저 저장/복원
  - 저장 키
  - 초기화
  - 저장 포맷 검증
- `data` 또는 도메인 계층
  - 저장소를 조합하는 사용자 데이터 조작
  - 문제 세트 저장/활성화 같은 흐름
- `quiz` / `dashboard`
  - 세션 해석, 결과 요약, 통계 요약 같은 파생 계산

핵심은 저장소가 곧바로 UI 의미를 갖지 않게 하고, 파생 계산이 저장 구현과 지나치게 결합되지 않게 만드는 것이다.

## Migration and Compatibility Notes

이번 리팩터는 아래 위험을 고려해야 한다.

- 저장 키를 바꾸면 기존 로컬 데이터가 사라진 것처럼 보일 수 있다.
- 파싱/검증 방식이 달라지면 기존 저장 데이터가 예상보다 자주 초기화될 수 있다.
- 구조를 과하게 일반화하면 오히려 디버깅이 어려워질 수 있다.

따라서 리팩터는 가능하면 아래 순서를 따른다.

1. 현재 저장 포맷 유지
2. 내부 파일 구조와 인터페이스만 정리
3. 필요할 때만 버전 증가나 마이그레이션 전략 검토

## Out of Scope

- PDF 업로드/검수 화면 구현
- 서버 동기화

## Dependency Notes

이 단계는 아래 브랜치/기능과 직접 연결된다.

- PDF 업로드/검수 흐름은 최종적으로 문제 세트 저장 구조에 의존한다.
- `feature/question-data`는 문제 세트 저장/활성화 인터페이스에 의존한다.
- `feature/localstorage-progress`는 현재 저장소 통합 상태를 제공한다.
- `feature/statistics-dashboard`는 저장 해석 계층 위에 올라간다.
- 이후 버그 수정 브랜치는 이 리팩터 결과를 기반으로 안정화될 가능성이 높다.

## Risks

- 리팩터 범위를 넓히면 기능 브랜치처럼 커져서 위험해질 수 있다.
- 구조 정리에만 몰두하면 실제 개선보다 파일 이동만 많아질 수 있다.
- 저장 포맷 변경을 섣불리 건드리면 기존 로컬 데이터 호환성이 깨질 수 있다.
- 지금 충분히 단순한 부분까지 과도하게 일반화하면 오히려 이해가 어려워질 수 있다.

## Notes

- 현재 저장 계층은 이미 기능적으로는 충분히 동작하고 있다.
- 따라서 이 브랜치는 "고장난 구조 수리"보다 "앞으로 복잡해질 구조를 미리 다듬는 작업"에 가깝다.
- PDF 업로드/검수 기능이 본격적으로 붙기 전에 저장 모델을 손볼 마지막 안전한 타이밍일 수 있다.
