# feature/pdf-upload-intake Plan

## Goal

사용자가 문제와 답이 담긴 PDF 파일을 앱에 가져올 수 있는 첫 번째 진입점을 만들고, 이후 변환/검수 단계가 소비할 수 있는 안전한 입력 상태를 준비한다.

## Branch Intent

이 브랜치는 "PDF 파일을 앱 내부 처리 흐름으로 받는 것"만 담당한다.

- 사용자가 홈 또는 전용 가져오기 화면에서 PDF 파일을 선택할 수 있게 한다.
- 업로드 가능한 형식과 기본 제약을 명확히 보여준다.
- 업로드된 파일을 이후 변환/검수 단계가 바로 소비할 수 있는 입력 상태로 전달한다.
- 아직 실제 문제 추출이나 저장 확정은 하지 않는다.

- [x] 사용자가 홈 또는 전용 가져오기 화면에서 PDF 파일을 선택할 수 있게 한다.
- [x] 업로드 가능한 형식과 기본 제약을 명확히 보여준다.
- [x] 업로드된 파일을 이후 변환/검수 단계가 바로 소비할 수 있는 입력 상태로 전달한다.
- [x] 아직 실제 문제 추출이나 저장 확정은 하지 않는다.

이번 단계의 핵심은 "PDF를 문제로 바꾸는 것"이 아니라, "PDF를 신뢰 가능한 입력으로 받아들이는 것"이다.

## Scope

- `/import` 전용 진입 화면
- 파일 선택 입력
- 필요 시 드래그 앤 드롭 영역
- PDF 형식 검증
- 용량/빈 파일 기본 검증
- 업로드 실패/성공 상태 표시
- 다음 단계로 넘길 최소 입력 상태 정의

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 사용자가 어떤 파일을 준비해야 하는지 이해할 수 있도록 설명 제공
- PDF 이외 형식을 차단하고, 왜 차단됐는지 안내하기
- 지나치게 큰 파일이나 비어 있는 파일을 기본 검증으로 막기
- 업로드된 파일을 메모리 상태, URL state, 또는 브라우저 임시 저장 중 어떤 방식으로 다음 단계에 넘길지 정리하기
- 이후 `feature/pdf-import-review`가 재사용할 수 있는 입력 타입과 전달 경로를 고정하기
- 업로드가 성공하면 사용자가 다음 단계가 "변환/검수"라는 것을 자연스럽게 이해하도록 흐름을 설계하기

이번 브랜치에서는 PDF 텍스트 파싱이나 문항 분리 알고리즘은 다루지 않는다. 입력 경험과 전달 책임만 다룬다.

## Intake Principles

PDF 업로드 진입은 아래 원칙을 따라야 한다.

- 사용자는 업로드 전에 지원 형식과 제한을 알아야 한다.
- 실패는 조용히 무시하지 말고, 이유를 짧고 명확하게 보여줘야 한다.
- 업로드 성공은 "파일을 받았다"와 "문제 추출이 끝났다"를 혼동하게 만들면 안 된다.
- 입력 상태는 다음 단계에서 재사용 가능해야 하며, 현재 화면 전용 상태로만 끝나면 안 된다.
- 아직 장기 저장이 아닌 임시 입력 상태라는 점을 구조에서 분명히 해야 한다.

## Proposed Intake State

이번 단계에서는 최소한 아래 수준의 입력 상태가 필요할 가능성이 높다.

- `selectedFileName: string`
- `selectedFileSize: number`
- `selectedFileType: string`
- `selectedFile: File`
- `errorMessage: string | null`
- `isReadyForReview: boolean`

필요 시 다음 단계 전달용 구조:

- `PdfImportInput`
  - `fileName`
  - `fileSize`
  - `fileType`
  - `file`
  - 필요 시 `objectUrl`

핵심은 "사용자가 고른 PDF 자체"와 "화면 표시용 정보"를 구분하는 것이다.

## Validation Rules

이번 단계의 파일 검증은 최소한 아래를 만족해야 한다.

- 파일이 존재해야 한다.
- MIME type 또는 파일 확장자 기준으로 PDF만 허용해야 한다.
- 파일 크기가 0이면 안 된다.
- 파일 크기가 지나치게 크면 제한해야 한다.
- 실패 시 에러 메시지를 즉시 보여줘야 한다.

이 단계의 검증은 보안 수준의 완전한 검증이 아니라, 다음 단계로 넘기기 전 기본적인 입력 품질을 보장하는 수준이면 충분하다.

## Transition Rules

업로드 후 다음 단계 전환 규칙은 아래를 따라야 한다.

- 유효한 PDF가 선택되기 전에는 다음 단계로 갈 수 없어야 한다.
- 유효한 PDF가 준비되면 검수/변환 단계로 이동할 수 있어야 한다.
- 다음 단계가 아직 미구현이어도, 최소한 "이 파일이 다음 단계 입력으로 준비됨"을 명확히 보여줄 수 있어야 한다.
- 이후 `feature/pdf-import-review`가 붙을 때 라우트나 상태 전달 구조를 다시 뒤엎지 않아도 되도록 설계해야 한다.

## Interaction Design

이번 단계에서 기대하는 상호작용은 다음과 같다.

### 1. Before Upload

- 사용자는 어떤 PDF를 올려야 하는지 설명을 읽는다.
- 허용 형식과 제한 조건을 확인할 수 있다.

### 2. File Selection

- 사용자는 파일 선택 버튼 또는 드래그 앤 드롭으로 PDF를 고른다.
- 잘못된 파일이면 즉시 오류 메시지를 본다.

### 3. Ready State

- 올바른 PDF를 고르면 파일 이름과 기본 정보가 표시된다.
- 사용자는 다음 단계로 갈 준비가 됐음을 이해한다.

### 4. Error State

- 선택한 파일이 잘못됐으면 왜 안 되는지 이해 가능한 메시지가 나온다.
- 사용자는 다시 파일을 고를 수 있어야 한다.

### 5. Non-goals in This Step

- PDF 미리보기 렌더링
- OCR
- 텍스트 추출
- 문제/보기/정답 자동 생성
- 검수 편집 UI

## UI Principles

- 업로드 영역은 명확한 시작점처럼 보여야 한다.
- 성공 상태와 오류 상태는 시각적으로 구분돼야 한다.
- 사용자는 "업로드 성공"과 "추출 완료"를 혼동하지 않아야 한다.
- 아직 가져오기 첫 단계이므로, 설명과 액션이 지나치게 많아지지 않는 편이 좋다.

## Data Flow Notes

현재 제품 기준선은 "저장된 활성 문제 세트" 중심이지만, 그 시작점은 PDF 가져오기다.

이 단계의 데이터 흐름은 아래처럼 해석할 수 있다.

1. 사용자가 `/import`에 들어온다.
2. PDF 파일을 선택한다.
3. 앱이 파일 유효성을 검증한다.
4. 유효한 파일을 다음 단계 입력 상태로 보관한다.
5. 이후 `feature/pdf-import-review`가 이 입력 상태를 받아 변환/검수 흐름을 시작한다.

즉, 이번 브랜치의 역할은 "문제 세트 저장"이 아니라 "문제 세트 생성 흐름의 시작점 확보"다.

- 문제 세트 저장 확정
- LocalStorage 영구 저장
- 서버 업로드

- [x] 사용자가 문제 PDF를 앱 안으로 가져오는 첫 진입점을 사용할 수 있다.
- [x] 파일 유효성 검사가 최소 수준으로 안정적으로 동작한다.
- [x] 이후 브랜치가 바로 변환/검수 단계로 이어질 수 있다.
- [x] `/import`가 더 이상 placeholder가 아니라 실제 시작점으로 동작한다.

## Risks

- 업로드와 저장을 한 단계에서 섞으면 책임 경계가 흐려질 수 있다.
- 입력 상태를 현재 화면 전용으로만 설계하면 다음 브랜치에서 재작업이 커질 수 있다.
- 오류 상태가 약하면 사용자는 "왜 업로드가 안 되는지" 이해하지 못할 수 있다.
- PDF를 이미 처리한 것처럼 보이게 만들면 이후 검수 단계 UX가 혼란스러워질 수 있다.

## Notes

- 이 단계는 PDF 처리의 시작점이지, 문제 데이터 완성 단계가 아니다.
- 아직 "문제를 바로 푼다"가 아니라 "문제를 가져온다"에 집중한다.
- 구현 시에는 브라우저 안에서만 다루는 임시 입력 상태와, 이후 저장될 확정 데이터 구조를 분리하는 것이 중요하다.
