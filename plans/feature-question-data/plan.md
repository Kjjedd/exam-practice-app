# feature/question-data Plan

## Goal

앱이 소비할 정적 문제 데이터를 준비하고, 이후 브랜치에서 재사용 가능한 데이터 접근 구조를 만든다.

## Branch Intent

이 브랜치는 "도메인 타입이 준비된 상태에서 실제 문제 데이터를 연결 가능한 형태로 만든다"는 목적만 담당한다.

- [x] `Question` 타입에 맞는 정적 문제 데이터를 만든다.
- [x] 이후 문제 UI와 채점 로직이 바로 사용할 수 있는 데이터 형태를 고정한다.
- [x] 문제 데이터 접근 방식을 단순하고 재사용 가능하게 정리한다.

이 단계에서는 데이터와 데이터 접근 구조만 다루고, 화면 동작이나 채점 로직은 구현하지 않는다.  
현재 브랜치에서는 이 원칙을 지킨 상태로 정적 문제 데이터와 데이터 로더만 구현했다.

## Scope

- `Question` 타입에 맞는 정적 문제 데이터 작성
- `questions.json` 샘플 데이터 구성
- 문제 ID, 보기, 정답, 해설 구조 일관성 확보
- 정적 JSON import 기반 데이터 로딩 방식 정리
- 이후 브랜치에서 재사용 가능한 데이터 로더 유틸 작성

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 샘플 문제 데이터의 최소 품질 기준 확정
- 문제 배열을 안전하게 불러오는 방식 정리
- 다음 브랜치가 문제 목록 전체와 단일 문제 조회를 모두 쉽게 사용할 수 있는 형태 준비
- 문제 데이터가 타입과 실제 사용 흐름을 동시에 만족하는지 검증

이번 브랜치에서는 "문제 데이터의 신뢰성"이 핵심이며, UI 편의를 위한 가공이나 저장소 구조는 넣지 않는다.

## Data Principles

문제 데이터는 아래 원칙을 따라야 한다.

- 모든 문제는 고유한 `id`를 가진다.
- 모든 문제는 `Question` 타입과 일치한다.
- `answer`는 반드시 `choices` 인덱스 범위 안에 있어야 한다.
- 모든 문제는 `explanation`을 포함한다.
- 최소한 샘플 데이터라도 실제 앱 흐름을 검증할 수 있을 정도로 완전해야 한다.
- 문제 데이터는 이후 랜덤 모드, 결과 집계, 오답 복습까지 견딜 수 있는 기본 품질을 가져야 한다.

## Proposed Data Shape

이번 단계에서는 이미 정의된 `Question` 타입을 기준으로 아래 형태를 따르는 것이 적절하다.

- `id`
- `question`
- `choices`
- `answer`
- `explanation`
- `category`

추가 필드는 지금 단계에서 섣불리 늘리지 않는다.  
현재 목표는 최소 필드로도 문제풀이 흐름을 자연스럽게 연결할 수 있는 기반을 만드는 것이다.

## Loader Strategy

데이터 로더 유틸은 최대한 단순해야 한다.

가능한 책임:

- 전체 문제 목록 반환
- 문제 ID 기반 단일 문제 조회 또는 조회에 필요한 기초 제공
- 이후 브랜치에서 재사용 가능한 import 경로 정리

이번 단계에서 피해야 하는 것:

- 과도한 캐싱 구조
- 필터링/셔플 같은 비즈니스 로직
- 저장소나 세션 상태와 결합된 처리

## Sample Data Expectations

샘플 문제는 단순 개수 채우기가 아니라, 실제 이후 브랜치 검증에 도움이 되어야 한다.

- 최소 5개 이상
- 동일한 구조 유지
- 보기 개수 일관성 유지
- 서로 다른 `id`
- 해설 포함
- 카테고리 값 포함

가능하다면 이후 화면 테스트에 도움이 되도록 문제 길이나 문장 스타일이 너무 비슷하지만은 않게 준비하는 편이 좋다.

## Out of Scope

- 문제풀이 UI
- 채점 로직
- 결과 계산 로직
- 저장/통계 기능 구현
- 즐겨찾기 기능 구현
- LocalStorage 구조
- 문제 선택 상태
- 결과 타입 변경

- [x] 최소 5개 이상의 샘플 문제 데이터

이번 단계에서 예상하는 파일 구조는 아래와 같다.

- `data/questions.json`
- `lib/data/load-questions.ts`
- 필요 시 `lib/data/index.ts`

## Dependency Notes

이 단계는 다음 브랜치들의 직접적인 기반이 된다.

- `feature/home-page`는 시작 버튼 이후 연결 가능한 실제 문제 데이터 존재를 전제로 한다.
- `feature/question-ui`는 렌더링할 문제 데이터가 필요하다.
- `feature/answer-checking`은 `answer` 값의 일관성에 직접 의존한다.
- `feature/result-summary`와 `feature/wrong-answer-review`는 문제 ID 기반 추적이 가능해야 안정적이다.

따라서 이번 단계에서 문제 데이터 품질이 낮으면 이후 브랜치들이 UI 문제처럼 보이는 데이터 문제를 계속 떠안게 된다.

## Validation

- [x] JSON 구조가 `Question` 타입과 일치해야 한다.
- [x] 모든 문제의 `id`는 고유해야 한다.
- [x] 모든 문제의 `choices`는 유효한 길이를 가져야 한다.
- [x] 모든 문제의 `answer` 값은 `choices` 범위 내에 있어야 한다.
- [x] 모든 문제는 `explanation`을 포함해야 한다.
- [x] 데이터 로더가 이후 브랜치에서 재사용 가능한 형태로 동작해야 한다.
- [x] `npm run typecheck` 기준으로 새 오류를 만들지 않아야 한다.

## Success Criteria

아래 조건을 만족하면 `feature/question-data` 단계가 완료된 것으로 본다.

- [x] 앱이 읽을 수 있는 정적 문제 JSON이 준비된다.
- [x] 문제 데이터가 `Question` 타입과 일치한다.
- [x] 로더 유틸이 다음 브랜치에서 별도 수정 없이 재사용 가능하다.
- [x] 샘플 문제 데이터만으로도 이후 문제 UI와 채점 브랜치를 진행할 수 있다.
- [x] 데이터 구조가 불필요하게 복잡하지 않다.

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- 문제 데이터에 저장소나 사용자 상태를 섞어 넣으면 책임이 흐려진다.
- 로더 유틸이 비즈니스 로직까지 맡으면 다음 브랜치에서 다시 분리가 필요해질 수 있다.
- `answer`와 `choices` 관계가 조금만 틀어져도 이후 채점 로직 전체가 불안정해진다.
- 샘플 데이터가 너무 빈약하면 이후 UI 브랜치에서 검증 가치가 떨어진다.

## Notes

- 데이터는 실제 앱이 소비할 수 있을 정도로 완전해야 한다.
- 하지만 지금 단계에서 과도한 데이터 모델 확장은 피한다.
- 필요하면 구현 전에 카테고리 분류 방식 정도는 짧게 설명할 수 있어야 한다.
- 이번 브랜치의 핵심은 "좋은 샘플 데이터와 단순한 접근 구조"다.
