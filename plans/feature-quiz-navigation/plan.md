# feature/quiz-navigation Plan

## Goal

현재 문제를 제출하고 해설을 확인한 뒤, 다음 문제로 자연스럽게 이동하고 마지막 문제에서는 세션 종료 흐름으로 넘어갈 수 있는 내비게이션 구조를 만든다.

## Branch Intent

이 브랜치는 "한 문제를 푸는 화면"을 "여러 문제를 순서대로 진행하는 세션 화면"으로 확장하는 단계다.

- [x] 현재 문제 인덱스를 기준으로 문제를 순차적으로 렌더링한다.
- [x] 제출과 해설 확인 이후에만 다음 문제로 이동할 수 있게 한다.
- [x] 마지막 문제에서는 다음 문제 이동 대신 세션 종료 흐름을 제공한다.
- [x] 다음 단계인 결과 화면 구현이 자연스럽게 올라갈 수 있도록 세션 종료 시점의 상태를 정리한다.

이번 단계의 핵심은 "이동"이지 "집계"가 아니다.  
즉, 결과를 계산하는 상세 화면은 아직 만들지 않지만, 결과 화면으로 넘어갈 수 있는 흐름은 준비해야 한다.

## Scope

- 현재 문제 인덱스 상태 도입
- 현재 문제를 인덱스 기준으로 렌더링
- 다음 문제 이동 버튼
- 제출 이후에만 이동 가능하도록 제어
- 마지막 문제에서 종료 버튼 또는 결과 보기 버튼 제공
- 결과 화면 전이를 위한 최소 라우팅 연결
- 문제 전환 시 선택/제출/해설 상태 초기화

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 지금까지 첫 문제 고정으로 렌더링하던 구조를 `currentQuestionIndex` 기반 구조로 변경
- `currentQuestion`를 `questions[currentQuestionIndex]`에서 가져오도록 수정
- 사용자가 제출과 해설 단계를 거친 뒤 다음 문제로 넘어가게 제어
- 다음 문제로 이동하면 이전 문제의 선택/제출/해설 상태가 현재 문제에 남지 않도록 초기화
- 마지막 문제에서는 "다음 문제"가 아니라 "결과 보기" 또는 동등한 종료 CTA를 제공
- 결과 화면 구현 전이라도 `/result`로 이동 가능한 최소 연결을 준비

이번 브랜치의 핵심은 "문항 간 이동 규칙을 확립하는 것"이며, 세션 전체의 정답률 계산까지 책임지지는 않는다.

## State Principles

세션 이동 상태는 아래 원칙을 따라야 한다.

- 현재 문제는 항상 하나의 인덱스로 표현되어야 한다.
- 인덱스는 문제 수 범위를 벗어나면 안 된다.
- 문제 전환 시 문제 단위 UI 상태는 초기화되어야 한다.
- 이동 가능 여부는 제출 상태에 종속되어야 한다.
- 마지막 문제 판정은 현재 인덱스와 전체 문제 수에 기반해 단순하게 계산되어야 한다.

## Proposed State Shape

이번 단계에서는 최소한 아래 수준의 상태 구조가 적절하다.

- `currentQuestionIndex: number`
- `currentQuestion: Question | undefined`
- `isSubmitted: boolean`
- `isCorrect: boolean | null`
- `isExplanationOpen: boolean`

그리고 유도 값으로 아래 정보가 필요할 가능성이 높다.

- `isLastQuestion: boolean`
- `currentQuestionNumber: number`
- `canGoToNextQuestion: boolean`

핵심은 문제 이동 상태와 문제 풀이 상태를 분리하되, 이동 가능 조건은 풀이 상태에 의존하도록 두는 것이다.

## Interaction Design

이번 단계에서 기대하는 상호작용은 다음과 같다.

### 1. Before Submit

- 사용자는 현재 문제를 읽고 답을 선택한다.
- 아직 다음 문제로 갈 수 없다.
- 이동 CTA가 보이더라도 비활성 상태이거나 숨겨져야 한다.

### 2. After Submit

- 사용자는 정답/오답과 해설을 확인한다.
- 다음 문제로 넘어갈 수 있는 CTA가 활성화된다.

### 3. Move To Next Question

- 사용자가 다음 문제 CTA를 누르면 다음 인덱스로 이동한다.
- 새 문제에서는 선택, 제출, 해설 상태가 모두 초기화된다.
- 진행 표시도 새 문제 번호에 맞게 갱신된다.

### 4. Last Question

- 마지막 문제에서는 다음 문제 CTA 대신 세션 종료 CTA가 나타난다.
- 이 CTA는 결과 화면 또는 종료 흐름으로 연결된다.

### 5. Non-goals in This Step

- 누적 정답 수 계산
- 문제별 결과 목록 작성
- 오답만 추리기
- 통계 저장

## UI Principles

- 이동 CTA는 제출/해설 흐름을 방해하지 않게 배치해야 한다.
- 사용자는 "지금 무엇을 할 수 있는지"를 명확히 이해할 수 있어야 한다.
- 마지막 문제에서는 CTA 문구가 분명히 달라져야 한다.
- 문제 전환 시 이전 문제의 피드백이 남아 있으면 안 된다.
- 진행 표시와 내비게이션은 서로 같은 현재 인덱스를 바라봐야 한다.

## Navigation Rules

세션 이동 규칙은 최소한 아래를 만족해야 한다.

- 현재 문제가 제출되기 전에는 다음 문제로 이동할 수 없다.
- 현재 문제가 마지막 문제가 아니면 다음 문제로 이동할 수 있다.
- 현재 문제가 마지막 문제이면 종료 흐름으로 전환해야 한다.
- 인덱스 증가 로직은 범위를 벗어나지 않도록 보호해야 한다.

## Result Transition Notes

이번 단계는 결과 화면 상세 구현 전 단계이지만, 전이 지점은 준비해야 한다.

- 마지막 문제 종료 CTA는 `/result`로 이동할 수 있어야 한다.
- 아직 결과 집계 데이터가 없더라도, 최소한 라우팅 구조는 끊기지 않아야 한다.
- 다음 단계 `feature/result-summary`에서 이 전이 지점 위에 집계와 요약 UI가 올라갈 수 있어야 한다.

## Data Usage Notes

이번 단계는 활성 문제 세트 전체 문제 배열을 실제로 순회하기 시작하는 첫 단계다.

- `questions[currentQuestionIndex]`를 현재 문제로 사용
- `questions.length`를 진행 수치와 마지막 문제 판정에 사용
- 활성 문제 세트가 없는 경우에는 기존 빈 상태를 그대로 유지

이 단계부터는 더 이상 "첫 문제 하나만 보여주는 화면"이 아니라 "문제 배열을 순서대로 진행하는 세션 화면"이 된다.

- 시험 모드 특화 흐름
- LocalStorage 기반 진행 저장

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- 인덱스 상태와 선택/제출/해설 상태가 어긋나면 잘못된 문제에 결과가 붙을 수 있다.
- 마지막 문제 판정이 불안정하면 결과 화면 진입이 깨질 수 있다.
- 문제 전환 시 상태 초기화가 누락되면 이전 문제의 정답/오답 UI가 다음 문제에 남을 수 있다.
- 결과 집계까지 한 번에 넣으려 하면 다음 브랜치 경계가 무너진다.

## Notes

- 이번 브랜치의 핵심은 "한 문제 화면"을 "문제 세션 화면"으로 바꾸는 것이다.
- 세션 이동 규칙이 안정되어야 이후 결과, 랜덤, 시험 모드가 자연스럽게 이어진다.
- 이 단계에서는 집계보다 이동 규칙의 명확성이 더 중요하다.
