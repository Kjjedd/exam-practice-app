# feature/answer-checking Plan

## Goal

사용자가 선택한 보기를 제출하고, 현재 문제에 대해 정답/오답을 즉시 판정하는 핵심 채점 흐름을 만든다.

## Branch Intent

이 브랜치는 "선택된 보기 위에 제출 시점과 정답 판정 결과를 얹되, 아직 해설 패널과 최종 결과 화면으로는 넘어가지 않는다"는 목적만 담당한다.

- [x] 선택된 보기가 있을 때만 제출 가능한 상태를 만든다.
- [x] 제출 시 현재 문제의 정답/오답을 정확하게 판정한다.
- [x] 제출 후에는 선택 상태를 고정하고 결과를 화면에 반영한다.
- [x] 다음 단계의 해설 노출과 결과 집계가 자연스럽게 올라갈 수 있도록 제출 결과 구조를 분명히 만든다.

이 단계는 퀴즈 화면에서 처음으로 "판정"이 일어나는 지점이다.  
하지만 해설 확장, 문제 이동, 세션 전체 결과 요약은 아직 하지 않는다.

## Scope

- 제출 버튼 UI
- 제출 가능 조건 처리
- 정답 판정 로직
- 제출 이후 상태 고정
- 현재 문제 기준 정답/오답 시각 표시
- 사용자 선택과 실제 정답의 차이 시각화
- 현재 문제에 대한 단일 제출 결과 구조

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- `feature/answer-selection`에서 만든 `selectedChoiceIndex`를 제출 입력으로 사용
- 선택이 없을 때는 제출 버튼을 비활성화
- 제출 시 현재 문제의 `answer`와 `selectedChoiceIndex`를 비교
- 제출 이후에는 선택 변경이 더 이상 일어나지 않도록 상호작용 고정
- 사용자가 고른 보기와 실제 정답 보기를 구분해서 표시
- 이후 `feature/explanation-panel`과 `feature/quiz-navigation`이 재사용할 수 있는 현재 문제 결과 상태 정리

이번 브랜치의 핵심은 "정답인지 아닌지 알 수 있다"는 경험을 만드는 것이지, "해설까지 다 본다"는 경험을 완성하는 것이 아니다.

## State Principles

채점 상태는 아래 원칙을 따라야 한다.

- 제출 전 상태와 제출 후 상태를 명확히 구분해야 한다.
- 제출 전에는 정답 정보가 절대 노출되면 안 된다.
- 제출 후에는 현재 문제에 대한 선택 변경이 막혀야 한다.
- 현재 단계의 결과 상태는 최소한 아래 정보를 설명할 수 있어야 한다.
  - 제출 여부
  - 선택한 보기
  - 실제 정답
  - 정답 여부
- 세션 전체 결과 배열이나 누적 기록 저장은 아직 하지 않는다.

## Proposed State Shape

이번 단계에서는 현재 문제에 대해 아래 수준의 상태가 적절하다.

- `selectedChoiceIndex: ChoiceIndex | null`
- `submittedChoiceIndex: ChoiceIndex | null`
- `isSubmitted: boolean`
- `isCorrect: boolean | null`

또는 이와 동등한 구조라면 가능하다.  
중요한 것은 선택 상태와 제출 결과 상태가 역할상 분리되어 있어야 한다는 점이다.

## Interaction Design

이번 단계에서 기대하는 상호작용은 다음과 같다.

### 1. Before Submit

- 선택된 보기가 없으면 제출 버튼 비활성
- 선택된 보기가 있으면 제출 버튼 활성
- 보기 선택은 자유롭게 변경 가능
- 정답/오답 관련 시각 정보는 표시되지 않음

### 2. Submit

- 사용자가 제출 버튼을 누르면 즉시 판정
- `selectedChoiceIndex`를 기준으로 정답 여부 계산
- 현재 문제가 제출 완료 상태로 바뀜

### 3. After Submit

- 보기 선택 변경 불가
- 사용자가 고른 보기와 실제 정답 보기가 구분되어 보임
- 제출 버튼은 비활성 또는 제출 완료 상태로 바뀜

### 4. Non-goals in This Step

- 해설 펼침/접힘 없음
- 다음 문제 이동 없음
- 결과 화면 이동 없음

## UI Principles

- 제출 버튼의 활성/비활성 기준이 명확해야 한다.
- 제출 후 결과는 즉시 읽을 수 있어야 한다.
- 사용자 선택과 정답 표시가 헷갈리지 않아야 한다.
- 아직 해설 단계가 아니므로 텍스트 과잉 설명보다 명확한 시각 구분이 우선이다.
- 정답/오답 색상 의미는 명확하되, 접근성을 고려해 색상에만 의존하지 않는 편이 좋다.

## Result Display Rules

제출 후 보기 목록은 최소한 아래 구분을 지원해야 한다.

- 사용자가 선택한 보기
- 실제 정답 보기
- 사용자가 선택했고 정답이기도 한 경우
- 사용자가 선택했지만 오답인 경우

즉, 제출 후에는 "내가 뭘 골랐는지"와 "무엇이 정답인지"를 모두 알 수 있어야 한다.

## Data Usage Notes

이번 단계는 여전히 활성 문제 세트의 첫 문제를 기준으로 동작한다.

- 현재 문제의 `answer` 값을 사용해 판정
- 현재 문제 결과만 화면 상태로 유지
- 활성 문제 세트가 없는 경우에는 기존 빈 상태를 그대로 유지

이번 브랜치에서는 세션 전체 결과 누적보다 "현재 문제 한 개의 제출과 판정"이 더 중요하다.

## Dependency Notes

이 단계는 아래 브랜치들과 직접 연결된다.

- `feature/answer-selection`에서 만든 `selectedChoiceIndex`에 직접 의존한다.
- `feature/explanation-panel`은 이번 제출 완료 상태 위에 해설 노출을 얹게 된다.
- `feature/quiz-navigation`은 제출 완료 상태를 보고 다음 문제 이동 가능 시점을 정하게 될 가능성이 크다.
- `feature/result-summary`는 이후 세션 결과 구조가 필요하겠지만, 이번 단계에서는 현재 문제 결과 표현까지만 다룬다.

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- 제출 상태와 해설 상태를 한 번에 섞으면 다음 브랜치 경계가 무너진다.
- 제출 후에도 선택 변경이 가능하면 결과 신뢰도가 깨진다.
- 정답 표시와 사용자 선택 표시가 섞이면 사용자가 결과를 오해할 수 있다.
- 세션 전체 결과 저장까지 욕심내면 현재 브랜치 범위를 넘게 된다.

## Notes

- 이번 브랜치의 핵심은 "선택한 답을 확정하고 판정받는 순간"을 만드는 것이다.
- 현재는 한 문제 기준으로만 생각하는 편이 안전하다.
- 결과 구조는 다음 브랜치에서 해설과 내비게이션이 붙을 수 있도록 너무 작지도, 너무 크지도 않게 유지하는 것이 중요하다.
