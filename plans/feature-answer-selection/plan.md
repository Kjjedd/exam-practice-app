# feature/answer-selection Plan

## Goal

현재 활성 문제에 대해 사용자가 보기 하나를 선택하고, 제출 전까지 자유롭게 바꿀 수 있는 상태 구조와 UI를 만든다.

## Branch Intent

이 브랜치는 "문제 화면 위에 단일 보기 선택 상태를 얹되, 아직 채점이나 제출 결과는 표시하지 않는다"는 목적만 담당한다.

- [x] 활성 문제에 대해 하나의 보기만 선택할 수 있는 상태를 만든다.
- [x] 선택된 보기가 시각적으로 구분되도록 UI를 확장한다.
- [x] 제출 전에는 선택을 바꿀 수 있도록 상호작용을 허용한다.
- [x] 다음 단계의 제출/채점 로직이 자연스럽게 올라갈 수 있도록 상태 경계를 분명히 유지한다.

이 단계에서는 보기 클릭과 선택 강조까지만 구현한다.  
정답 여부, 제출 완료 상태, 결과 기록 생성은 다음 브랜치로 넘긴다.

## Scope

- 단일 선택 상태 관리
- 보기 클릭/탭 상호작용
- 선택된 보기 강조 UI
- 선택 전/선택 후 시각 차이
- 선택 상태를 현재 문제 기준으로만 유지하는 구조
- 모바일/데스크톱 터치 영역 가독성 개선

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- `feature/question-ui`에서 만든 읽기 전용 보기 목록을 상호작용 가능한 구조로 확장
- 현재 활성 문제에 대해 `selectedChoiceIndex` 같은 명확한 상태 도입
- 아직 제출하지 않은 상태에서는 클릭으로 선택을 여러 번 바꿀 수 있게 설계
- 선택되지 않은 보기와 선택된 보기를 시각적으로 확실히 구분
- 이후 `feature/answer-checking`에서 제출 이후 상태를 별도 분기로 얹을 수 있도록 상태 책임을 분리

이번 브랜치의 핵심은 "선택할 수 있다"는 경험을 만드는 것이지, "정답을 판정한다"는 경험을 만드는 것이 아니다.

## State Principles

선택 상태는 아래 원칙을 따라야 한다.

- 한 문제에 대해 동시에 하나의 보기만 선택 가능해야 한다.
- 아직 제출하지 않은 상태에서는 선택 변경이 가능해야 한다.
- 선택 상태는 현재 보이는 문제와 직접 연결되어야 한다.
- 제출 여부나 정답 여부와 같은 채점 상태를 미리 섞어 넣지 않는다.
- 선택 상태는 이후 문제 이동이 생기더라도 문제 ID 또는 현재 인덱스 기준으로 확장 가능해야 한다.

## Interaction Design

이번 단계에서 기대하는 상호작용은 다음과 같다.

### 1. Initial State

- 어떤 보기도 선택되지 않은 상태
- 시각적으로 모든 보기의 중요도가 동일하게 보임

### 2. Selected State

- 사용자가 보기를 클릭하거나 탭하면 해당 보기 하나만 선택됨
- 선택된 보기에는 배경, 테두리, 라벨 강조 등 분명한 변화가 있어야 함

### 3. Re-selection

- 사용자가 다른 보기를 누르면 이전 선택은 해제되고 새 선택으로 교체됨
- 아직 제출 단계가 아니므로 경고나 제한 없이 자연스럽게 바뀌어야 함

### 4. Non-goals in This Step

- 선택 취소 토글을 꼭 지원할 필요는 없음
- 정답/오답 색상 표시 금지
- 제출 버튼 동작 금지

## UI Principles

- 선택된 보기는 한눈에 알아볼 수 있어야 한다.
- 선택되지 않은 보기와 선택된 보기의 대비가 충분해야 한다.
- 모바일에서도 눌리기 쉬운 영역 크기를 유지해야 한다.
- 키보드 접근을 고려하면 이후 확장이 쉬워진다.
- 아직 채점 전이므로 성공/실패처럼 읽히는 색상 의미는 피하는 편이 안전하다.

## Data Usage Notes

이번 단계는 여전히 활성 문제 세트의 첫 문제를 기준으로 동작한다.

- `feature/question-ui`에서 연결된 현재 문제 데이터를 그대로 사용
- 선택 상태는 현재 문제에 대해 별도로 유지
- 활성 문제 세트가 없는 경우에는 기존 빈 상태 흐름을 그대로 유지

이번 브랜치에서는 여러 문제에 대한 누적 선택 기록보다 "현재 문제에 대한 단일 선택 경험"이 더 중요하다.

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- 선택 상태와 제출 상태를 한 번에 넣으면 다음 브랜치 경계가 무너진다.
- 선택 강조가 너무 약하면 사용자가 현재 선택을 인지하기 어렵다.
- 반대로 선택 강조가 정답/오답처럼 보이면 의미가 섞인다.
- 상태를 현재 문제가 아니라 화면 전체에 느슨하게 두면 이후 문제 이동 시 구조가 흔들릴 수 있다.

## Notes

- 다음 브랜치가 제출 시점을 분명히 다룰 수 있도록 상태 이름과 책임을 단순하게 유지하는 것이 중요하다.

## Instant Checking Revision

이후 수정에서는 제출 버튼을 제거하고, 보기 클릭 자체가 곧 답안 확정과 판정으로 이어지는 흐름으로 전환해야 한다.

- 선택 상태는 더 이상 "제출 전 임시 선택"만 의미하지 않는다.
- 클릭 즉시 판정 구조에서는 보기 클릭이 현재 문제의 답안 확정 이벤트가 된다.
- 따라서 미제출 상태에서의 자유로운 재선택 경험은 현재 문제를 아직 풀지 않은 경우에만 의미가 있고, 한 번 판정된 문제에서는 다시 클릭으로 상태를 바꾸면 안 된다.
- 이미 판정된 문제를 다시 방문했을 때는 저장된 결과를 읽기 전용 상태로 보여주는 쪽이 안전하다.

### Instant Selection Revision Deliverables

- [x] 보기 클릭 시 별도 제출 단계 없이 즉시 판정 흐름과 연결되는 선택 이벤트 구조
- [x] 아직 판정되지 않은 문제에서만 클릭 가능하도록 제한
- [x] 이미 판정된 문제 재방문 시 읽기 전용 상태 유지
- [x] 제출 버튼 전제 문구와 책임 제거

### Instant Selection Revision Validation

- [x] 보기 클릭 즉시 현재 문제 결과가 확정되어야 한다.
- [x] 한 번 판정된 문제에서는 추가 클릭으로 결과가 바뀌면 안 된다.
- [x] 아직 판정되지 않은 문제만 상호작용 가능해야 한다.
- [x] 제출 버튼이 사라져도 선택 흐름이 사용자에게 자연스럽게 이해되어야 한다.

## Visual Highlight Revision

이후 수정에서는 보기 상태가 사용자가 한눈에 결과를 이해할 수 있도록 더 선명하게 하이라이트되어야 한다.

- 아직 판정되지 않은 문제에서 보기 위를 선택할 때는 "선택됨" 상태가 분명히 보여야 한다.
- 이 선택 강조는 정답/오답 색상과 혼동되지 않도록 중립적인 강조여야 한다.
- 판정이 끝난 뒤에는 선택 전 강조 규칙보다 결과 하이라이트 규칙이 우선해야 한다.

### Visual Highlight Deliverables

- [x] 판정 후 하이라이트와 충돌하지 않는 선택 상태 표현 유지

- [x] 판정 이후에는 선택 강조보다 결과 하이라이트가 우선 표시되어야 한다.
