# feature/localstorage-progress Plan

## Goal

학습 진행 상태, 최신 풀이 세션, 즐겨찾기 상태, 그리고 사용자가 확정한 문제 세트 정보를 브라우저 저장소에서 일관되게 유지하고 복원할 수 있게 한다.

## Branch Intent

이 브랜치는 이미 개별적으로 생겨난 저장 로직들을 "하나의 저장 계층" 관점으로 정리하는 단계다.

- [x] 현재 분산된 `question bank`, `latest quiz session`, `favorites` 저장 책임을 명확히 구분한다.
- [x] 새로고침 이후에도 사용자가 마지막 학습 문맥을 자연스럽게 이어갈 수 있는 복원 기준을 정리한다.
- [x] 저장 데이터가 깨졌거나, 문제 세트가 바뀌었거나, 버전이 달라진 상황에서도 안전한 기본값을 보장한다.
- [x] 이후 `feature/statistics-dashboard`가 신뢰할 수 있는 저장 계층 위에서 동작하도록 기반을 안정화한다.

이번 단계의 핵심은 "새 LocalStorage 기능 하나를 추가하는 것"이 아니라, 현재 앱 전반의 사용자 데이터를 저장소 관점에서 통합하고 정리하는 것이다.

## Scope

- 활성 문제 세트 저장/복원 규칙 정리
- 최신 풀이 세션 저장/복원 규칙 정리
- 즐겨찾기 저장/복원 규칙 정리
- 공통 초기화 또는 리셋 흐름 정의
- 파싱 실패/불완전 데이터/버전 불일치에 대한 안전한 복구
- 저장 계층 간 책임 경계 정리

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 문제 세트 저장소가 "활성 문제 세트"와 "저장된 문제 세트 목록"을 책임진다는 점을 명확히 하기
- 최신 풀이 세션 저장소가 "가장 최근에 끝난 또는 이어질 수 있는 세션 상태"만 담당하게 하기
- 즐겨찾기 저장소가 `questionId` 목록만 저장하도록 유지하기
- 저장 키, 기본값, 파싱 규칙, 초기화 규칙을 유틸 수준에서 정리하기
- 화면 단에서 필요한 복원 흐름을 구현할 수 있도록 읽기/쓰기/초기화 인터페이스를 단순하게 유지하기
- 문제 세트가 비활성화되거나 교체된 상황에서도 세션/즐겨찾기 해석이 어떻게 안전하게 동작할지 기준을 세우기

이번 브랜치에서는 아직 통계 대시보드 UI를 만들지 않는다. 대신 통계가 신뢰할 수 있는 저장 기반을 갖추도록 준비한다.

## Storage Principles

저장 계층은 아래 원칙을 따라야 한다.

- 저장 책임은 도메인별로 분리한다.
- `question bank`, `quiz session`, `favorites`는 서로 다른 저장 단위로 다룬다.
- 저장 포맷은 가능한 단순하고 버전 가능한 구조여야 한다.
- 읽기 시 실패하면 throw보다 안전한 기본값을 우선한다.
- 쓰기 시에는 반드시 검증된 데이터만 저장한다.
- 화면 로직은 저장소 내부 구조를 직접 알지 않고, 유틸 인터페이스만 사용해야 한다.
- 초기화는 부분 초기화와 전체 초기화를 구분할 수 있어야 한다.

## Current Storage Context

현재 프로젝트에는 이미 아래 저장 흐름이 부분적으로 존재한다.

- `question bank`
  - 저장된 문제 세트 목록
  - 활성 문제 세트 ID
- `latest quiz session`
  - 가장 최근 풀이 세션
  - 세션 결과와 문제 순서
- `favorites`
  - 즐겨찾기한 `questionId` 목록

이번 브랜치는 이 세 저장소를 새로 발명하는 것이 아니라, 다음 구현 단계와 유지보수 단계에서 흔들리지 않도록 역할과 인터페이스를 정리하는 단계다.

## Proposed Storage Responsibilities

### 1. Question Bank

- 사용자가 확정해 저장한 문제 세트 목록을 유지한다.
- 현재 활성 문제 세트를 식별한다.
- 문제 세트 검증과 활성 세트 존재 여부를 보장한다.

### 2. Latest Quiz Session

- 가장 최근 풀이 세션을 저장한다.
- 세션 모드, 문제 순서, 결과, 시작/종료 시각을 유지한다.
- 결과 화면, 오답 복습, 통계 집계의 입력 데이터가 된다.

### 3. Favorites

- 사용자가 보관하고 싶은 문제의 `questionId` 목록만 저장한다.
- 문제 세트 자체나 결과 정보는 중복 저장하지 않는다.

### 4. Reset Utilities

- 필요 시 최신 세션만 초기화할 수 있어야 한다.
- 필요 시 문제 세트 저장소만 초기화할 수 있어야 한다.
- 필요 시 즐겨찾기만 초기화할 수 있어야 한다.
- 필요 시 전체 사용자 학습 데이터를 한 번에 초기화할 수 있는 경로를 제공할 수 있어야 한다.

## Proposed Data Shape

이번 단계에서는 저장 구조를 아래처럼 해석하는 것이 적절하다.

- `QuestionBank`
  - `version`
  - `activeQuestionSetId`
  - `questionSets`
- `QuizSession`
  - `mode`
  - `questionSetId`
  - `questionSetTitle`
  - `questionIds`
  - `currentQuestionIndex`
  - `results`
  - `startedAt`
  - `completedAt`
- `favoriteQuestionIds`
  - `QuestionId[]`

핵심은 "저장 데이터는 최소한으로 유지하고, 화면에서 필요한 해석은 읽은 뒤 조합한다"는 점이다.

## Restore Rules

복원 규칙은 최소한 아래를 만족해야 한다.

- 활성 문제 세트가 없으면 퀴즈 화면은 안전한 빈 상태로 돌아가야 한다.
- 최신 세션이 깨졌거나 문제 세트와 맞지 않으면 결과/복습 화면은 빈 상태 또는 안전한 안내를 보여야 한다.
- 즐겨찾기는 현재 활성 문제 세트에서 찾을 수 없는 문제를 무시하되, 저장소 자체는 깨지지 않아야 한다.
- 저장 포맷이 부분적으로 비어 있어도 기본값으로 복구 가능해야 한다.
- 저장소 버전이 맞지 않으면 무리하게 사용하지 않고 기본 구조로 되돌아가야 한다.

## Reset and Cleanup Rules

이번 단계에서 초기화 전략도 함께 정리해야 한다.

- 전체 초기화는 아래 상태를 모두 비운다.
  - 저장된 문제 세트
  - 활성 문제 세트 ID
  - 최신 세션
  - 즐겨찾기
- 부분 초기화는 저장 책임 단위별로 나뉘어야 한다.
- 리셋 후 화면은 깨진 상태가 아니라, 새 사용자가 처음 들어온 것 같은 안전한 상태가 되어야 한다.

## Interaction Design

이번 단계는 저장 계층 중심 브랜치이므로 UI보다 흐름이 중요하다.

### 1. Save Flow

- 문제 세트를 저장하면 활성 문제 세트까지 자연스럽게 반영될 수 있어야 한다.
- 퀴즈를 마치면 최신 세션이 저장되어 결과와 복습 화면에서 사용할 수 있어야 한다.
- 즐겨찾기 토글은 즉시 저장소에 반영되어야 한다.

### 2. Restore Flow

- 홈 화면은 활성 문제 세트 존재 여부를 읽어 진입 상태를 결정한다.
- 퀴즈 화면은 활성 문제 세트를 읽어 문제 렌더링 여부를 결정한다.
- 결과/복습 화면은 최신 세션을 읽어 화면 상태를 결정한다.
- 즐겨찾기 화면은 즐겨찾기 ID와 활성 문제 세트를 조합해 목록을 만든다.

### 3. Reset Flow

- 저장 데이터가 초기화되면 관련 화면은 빈 상태로 안전하게 전환되어야 한다.

## UI Principles

- 이번 브랜치의 중심은 저장 계층이므로 UI는 최소 수준이어야 한다.
- 초기화 또는 복원 상태를 노출한다면, 사용자가 데이터 손실 여부를 쉽게 이해할 수 있어야 한다.
- 저장소 문제 때문에 화면이 깨지는 대신, 이해 가능한 빈 상태가 우선되어야 한다.

## Data Flow Notes

현재 구조를 기준으로 보면 저장 흐름은 아래처럼 이어진다.

1. PDF 검수 완료 후 문제 세트를 저장한다.
2. 저장된 문제 세트 중 하나를 활성 문제 세트로 둔다.
3. 퀴즈 화면은 활성 문제 세트를 읽어 세션을 시작한다.
4. 세션 완료 후 결과를 최신 세션 저장소에 기록한다.
5. 결과/복습/통계는 최신 세션을 읽어 후속 화면을 구성한다.
6. 즐겨찾기는 위 흐름과 별도로 `questionId` 기준 개인 보관함으로 유지된다.

즉, 이 브랜치의 핵심은 각 기능이 제각각 저장하는 것이 아니라, 앱 전체에서 일관된 저장 흐름을 가지게 만드는 것이다.

## Risks

- 저장 책임을 하나의 거대한 유틸에 몰아넣으면 구조가 다시 흐려질 수 있다.
- 초기화 범위가 불명확하면 사용자가 의도하지 않은 데이터 손실을 겪을 수 있다.
- 문제 세트와 세션 결합이 지나치면 문제 세트 변경 시 복원 실패가 잦아질 수 있다.
- 저장 규칙을 느슨하게 두면 대시보드 집계 단계에서 불일치가 커질 수 있다.

## Notes

- 현재 프로젝트는 이미 일부 저장 기능을 갖고 있으므로, 이번 브랜치는 "첫 저장 기능 구현"이 아니라 "저장 계층 정리와 강화"에 가깝다.
- PDF 업로드 기반 제품에서는 문제 세트 자체가 사용자 데이터이므로, 문제 세트 저장은 단순 설정이 아니라 핵심 학습 자산 저장이다.
- 통계 기능보다 먼저 저장 계층을 안정화하는 것이 맞다.
