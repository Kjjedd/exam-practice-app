# feature/question-data Plan

## Goal

검수 완료된 문제 세트를 저장하고, 이후 브랜치가 활성 문제 세트를 일관되게 읽을 수 있는 데이터 접근 구조를 만든다.

## Branch Intent

이 브랜치는 "검수 완료된 문제 세트를 영구 저장하고, 앱이 소비할 활성 문제 데이터로 노출한다"는 목적만 담당한다.

- [x] 검수 완료된 문제 세트를 저장 가능한 구조로 정리한다.
- [x] 현재 활성 문제 세트를 읽는 접근 방식을 단순하고 재사용 가능하게 만든다.
- [x] 이후 문제 UI와 채점 로직이 저장된 문제 세트를 바로 사용할 수 있게 한다.

이 단계에서는 저장된 문제 세트와 데이터 접근 구조만 다루고, 문제풀이 화면 동작은 구현하지 않는다.  
기존의 정적 샘플 JSON 중심 구현은 초기 검증용 결과였으며, 현재 계획에서는 PDF 검수 후 저장 구조로 대체된다.

## Scope

- 검수 완료된 문제 세트 저장 구조 정의
- 활성 문제 세트 메타데이터 정의
- 저장된 문제 세트 읽기/쓰기 인터페이스 정리
- 문제 ID, 보기, 정답, 해설 구조 일관성 확보
- 이후 브랜치에서 재사용 가능한 데이터 로더 유틸 작성

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 검수 완료된 문제 세트를 브라우저에 안전하게 저장하는 방식 정리
- 활성 문제 세트가 없을 때의 기본 동작 정의
- 다음 브랜치가 문제 목록 전체와 단일 문제 조회를 쉽게 사용할 수 있는 접근 함수 준비
- 저장된 데이터가 타입과 실제 사용 흐름을 동시에 만족하는지 검증

이번 브랜치에서는 "저장된 문제 세트의 신뢰성"이 핵심이며, UI 편의를 위한 화면 구현은 넣지 않는다.

## Data Principles

저장되는 문제 세트는 아래 원칙을 따라야 한다.

- 모든 문제는 고유한 `id`를 가진다.
- 모든 문제는 `Question` 타입과 일치한다.
- `answer`는 반드시 `choices` 인덱스 범위 안에 있어야 한다.
- 저장이 끝난 세트는 이후 문제풀이에서 추가 정규화 없이 사용할 수 있어야 한다.
- 활성 문제 세트와 과거 저장 세트의 경계를 구분할 수 있어야 한다.
- 문제 데이터는 이후 랜덤 모드, 결과 집계, 오답 복습까지 견딜 수 있는 기본 품질을 가져야 한다.

## Proposed Data Shape

이번 단계에서는 이미 정의된 `Question` 타입을 기준으로, 저장 단위는 아래 두 층으로 나뉘는 것이 적절하다.

1. 단일 문제 레코드
2. 문제 세트 레코드

문제 세트 레코드는 최소한 아래 정보를 가져야 한다.

- 세트 `id`
- 세트 이름 또는 업로드 출처 메타데이터
- 생성 시각
- 문제 배열
- 현재 활성 세트 여부 또는 활성 세트를 가리킬 수 있는 참조 정보

지금 목표는 저장 단위를 만들되, 이후 문제풀이가 불필요한 래핑 없이 `Question[]`를 읽을 수 있는 구조를 만드는 것이다.

## Loader Strategy

데이터 접근 유틸은 최대한 단순해야 한다.

가능한 책임:

- 현재 활성 문제 세트 조회
- 활성 세트의 전체 문제 목록 반환
- 문제 ID 기반 단일 문제 조회 또는 조회에 필요한 기초 제공
- 검수 완료 데이터 저장
- 이후 브랜치에서 재사용 가능한 import 경로 정리

이번 단계에서 피해야 하는 것:

- 과도한 캐싱 구조
- 필터링/셔플 같은 비즈니스 로직
- 저장소와 세션 상태를 한 함수에 뒤섞는 처리

## Out of Scope

- 문제풀이 UI
- 채점 로직
- 결과 계산 로직
- 업로드/검수 화면 구현
- 통계 기능 구현
- 즐겨찾기 기능 구현
- 문제 선택 상태
- 결과 타입 변경

- [x] 검수 완료 데이터 저장 인터페이스

## Dependency Notes

이 단계는 다음 브랜치들의 직접적인 기반이 된다.

- `feature/home-page`는 활성 문제 세트 유무에 따라 시작/가져오기 CTA를 다르게 보여줄 수 있다.
- `feature/question-ui`는 렌더링할 활성 문제 데이터가 필요하다.
- `feature/answer-checking`은 `answer` 값의 일관성에 직접 의존한다.
- `feature/result-summary`와 `feature/wrong-answer-review`는 문제 ID 기반 추적이 가능해야 안정적이다.

따라서 이번 단계에서 저장된 문제 세트 품질이 낮으면 이후 브랜치들이 UI 문제처럼 보이는 데이터 문제를 계속 떠안게 된다.

## Validation

- [x] 저장된 문제 세트 구조가 `Question` 타입과 일치하는 문제 배열을 포함해야 한다.
- [x] 모든 문제의 `id`는 고유해야 한다.
- [x] 모든 문제의 `choices`는 유효한 길이를 가져야 한다.
- [x] 모든 문제의 `answer` 값은 `choices` 범위 내에 있어야 한다.
- [x] 활성 세트 조회 로더가 이후 브랜치에서 재사용 가능한 형태로 동작해야 한다.
- [x] 활성 세트가 없을 때도 예측 가능한 기본값 또는 상태를 반환해야 한다.
- [x] `npm run typecheck` 기준으로 새 오류를 만들지 않아야 한다.

## Success Criteria

아래 조건을 만족하면 `feature/question-data` 단계가 완료된 것으로 본다.

- [x] 앱이 읽을 수 있는 활성 문제 세트 저장 구조가 준비된다.
- [x] 저장된 문제 데이터가 `Question` 타입과 일치한다.
- [x] 로더 유틸이 다음 브랜치에서 별도 수정 없이 재사용 가능하다.
- [x] 저장된 활성 문제 세트만으로 이후 문제 UI와 채점 브랜치를 진행할 수 있다.
- [x] 데이터 구조가 불필요하게 복잡하지 않다.

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- 문제 데이터에 사용자 세션 상태까지 섞어 넣으면 책임이 흐려진다.
- 로더 유틸이 비즈니스 로직까지 맡으면 다음 브랜치에서 다시 분리가 필요해질 수 있다.
- `answer`와 `choices` 관계가 조금만 틀어져도 이후 채점 로직 전체가 불안정해진다.
- 활성 세트 개념을 느슨하게 만들면 홈/퀴즈/결과 흐름이 어느 데이터를 기준으로 동작하는지 모호해진다.

## Notes

- 데이터는 실제 앱이 소비할 수 있을 정도로 완전해야 한다.
- 하지만 지금 단계에서 과도한 데이터 모델 확장은 피한다.
- 문제 세트 메타데이터는 저장/불러오기와 활성 세트 구분에 필요한 수준만 유지한다.
- 이번 브랜치의 핵심은 "검수 완료 데이터의 안전한 저장과 단순한 접근 구조"다.
