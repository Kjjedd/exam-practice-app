# ExamMate Research Notes

## 1. 분석 대상과 해석 범위

이 폴더에는 실행 가능한 소스코드가 아니라, 프로젝트 구현을 지시하는 단일 기준 문서 `codex-project-guidelines-exam-webapp.md`가 존재한다.  
따라서 본 문서는 "현재 코드가 어떻게 동작하는가"를 설명하는 문서가 아니라, "이 프로젝트가 어떤 구조와 규칙으로 구현되어야 하며, 완성된 앱은 어떤 방식으로 동작해야 하는가"를 재구성한 분석 문서다.

핵심 전제는 다음과 같다.

- 이 프로젝트는 개인용 시험 대비 예상문제풀이 웹앱이다.
- 서버 없는 정적 웹앱을 우선한다.
- 문제 데이터는 구조화된 JSON으로 관리한다.
- 학습 상태는 브라우저 `LocalStorage`에 저장한다.
- 구현은 Next.js + TypeScript + Tailwind CSS를 기반으로 한다.
- 모든 핵심 기능은 "옵션"이 아니라 필수 범위다.

즉, 이 프로젝트의 중심은 "문제풀이 경험"이고, 그 주변에 복습, 시험 모드, 즐겨찾기, 통계, 저장 기능이 계층적으로 확장되는 구조다.

## 2. 이 프로젝트의 본질

이 앱은 일반적인 콘텐츠 사이트가 아니라, 상태 전이가 분명한 학습용 인터랙티브 애플리케이션이다.  
사용자는 단순히 문제를 읽는 것이 아니라, 아래 흐름을 반복한다.

1. 문제 세트를 선택 또는 시작한다.
2. 현재 문제를 확인한다.
3. 보기 하나를 선택한다.
4. 제출한다.
5. 즉시 채점 결과를 본다.
6. 해설을 확인한다.
7. 다음 문제로 이동한다.
8. 전체 완료 후 결과를 확인한다.
9. 틀린 문제를 다시 복습하거나 다른 모드로 재도전한다.

즉, 핵심 도메인은 단순한 "문항 표시"가 아니라 다음 4개의 도메인으로 나뉜다.

- 문제 데이터 도메인
- 풀이 세션 도메인
- 학습 기록 도메인
- 개인화 기능 도메인(즐겨찾기, 통계, 최근 상태)

이 구조를 기준으로 보면, 앱의 동작은 사실상 "문제 데이터 + 세션 상태 머신 + 영속 저장소"의 조합으로 설명할 수 있다.

## 3. 문서가 정의하는 제품 목표

원문에서 가장 중요한 목표는 "모바일과 데스크톱 모두에서 접근 가능한 반응형 문제풀이 웹앱"이다.  
단순히 문제를 푸는 것만으로 끝나지 않고, 아래 경험을 하나의 제품 안에서 닫힌 루프로 제공해야 한다.

- 즉시 피드백
- 해설 기반 이해
- 결과 요약
- 오답 재학습
- 랜덤 재도전
- 시험형 연속 풀이
- 즐겨찾기 관리
- 누적 통계 확인
- 새로고침 이후 상태 유지

여기서 중요한 점은 문서가 MVP를 언급하면서도 실제로는 기능 축소를 허용하지 않는다는 것이다.  
즉, "초기에는 가볍게 시작하되 결국 문서에 적힌 기능은 모두 구현 대상"이라는 해석이 맞다.

## 4. 기술적 성격과 암묵적 제약

문서에서 직접 명시된 기술적 방향은 단순하지만, 실제로는 몇 가지 강한 제약을 만든다.

### 4.1 프론트엔드 중심 구조

- Next.js App Router 우선
- 서버 없는 정적 웹앱 중심
- 로컬 저장소 기반 상태 유지

이 조합은 다음을 의미한다.

- 초기 구현은 서버 API 없이도 완결되어야 한다.
- 핵심 데이터는 빌드 타임 또는 정적 파일로 제공될 가능성이 높다.
- 사용자의 풀이 이력은 브라우저마다 독립적으로 저장된다.
- 로그인, 동기화, 멀티 디바이스 연동은 현재 범위 밖이다.

### 4.2 데이터는 JSON 우선

PDF 원본 문제집을 런타임에서 직접 소비하지 않는다는 조건은 매우 중요하다.  
즉, 이 앱의 실행 흐름은 PDF 렌더링 앱이 아니라 정규화된 문제 레코드를 소비하는 퀴즈 엔진 쪽에 가깝다.

이는 아래 책임 분리를 강하게 시사한다.

- 데이터 변환 단계: PDF -> JSON
- 앱 실행 단계: JSON -> UI 렌더링 및 채점

현재 범위는 두 번째에 집중되어 있다.

### 4.3 상태 관리의 복잡도는 점진적

문서에서는 초기에는 React 상태 관리, 필요 시 Context 확장이라고 말한다.  
이는 전역 상태 라이브러리 도입보다 다음 순서를 선호한다는 의미다.

1. 로컬 컴포넌트 상태
2. 화면 단위 공유 상태
3. Context 기반 확장
4. 이후 필요 시 더 큰 상태 관리 도구 검토

즉, 과도한 추상화나 조기 아키텍처 확장을 경계한다.

## 5. 앱을 구성하는 핵심 기능 블록

문서를 기능 관점으로 해체하면 앱은 최소한 아래 블록으로 나뉜다.

### 5.1 문제풀이 블록

가장 중심이 되는 블록이다.

- 문제 번호 표시
- 문제 본문 표시
- 보기 렌더링
- 보기 선택 상태 유지
- 제출 가능 여부 판단
- 정답 판정
- 해설 노출
- 다음 문제 이동

이 블록이 정상 동작하지 않으면 나머지 기능은 의미가 없다.

### 5.2 결과 블록

문제풀이 세션이 종료된 뒤 생성되는 요약 계층이다.

- 총 문제 수
- 정답 수
- 오답 수
- 정답률
- 필요 시 미응답 수
- 정답/오답 구분 목록

즉, 결과 화면은 단순 통계 카드가 아니라 이후 복습 흐름으로 진입시키는 허브다.

### 5.3 복습 블록

오답 학습을 중심으로 하는 블록이다.

- 틀린 문제 목록 조회
- 각 문제의 정답/해설 재확인
- 오답 문제만 다시 풀기

이 블록은 학습 앱의 핵심 가치와 직접 연결된다.  
단순 퀴즈 앱과 학습 앱의 차이가 여기서 드러난다.

### 5.4 모드 블록

같은 문제 데이터라도 풀이 규칙과 UX를 다르게 만드는 기능이다.

- 랜덤 모드
- 시험 모드

랜덤 모드는 문제 순서를 바꾸는 기능이고, 시험 모드는 문제를 푸는 경험 자체를 더 연속적이고 평가 중심으로 만드는 기능이다.

### 5.5 개인화 블록

- 즐겨찾기
- 최근 학습 정보
- 누적 학습 통계
- 저장 상태 복원

이 블록은 사용자의 반복 학습을 지원하며, 앱이 단발성 사용에서 벗어나게 만든다.

## 6. 사용자 경험 흐름 재구성

문서 내용을 바탕으로 실제 앱 흐름을 재구성하면 다음과 같다.

### 6.1 메인 화면

메인 화면은 단순 랜딩 페이지가 아니라 학습 진입 허브가 되어야 한다.

가능한 주요 액션:

- 일반 문제풀이 시작
- 랜덤 모드 시작
- 시험 모드 시작
- 오답 복습 진입
- 즐겨찾기 문제 보기
- 통계 대시보드 보기
- 최근 학습 이어하기

문서에 메인 화면의 상세 UI는 없지만, 전체 기능 집합상 이런 분기 허브 역할이 자연스럽다.

### 6.2 문제풀이 화면

이 화면은 상태가 가장 많다.

- 현재 문제 인덱스
- 현재 문제 데이터
- 선택한 보기
- 제출 여부
- 채점 결과
- 해설 펼침 여부
- 현재 모드(일반/랜덤/시험/오답 재풀이)

동작 순서는 보통 다음과 같다.

1. 현재 문제 로드
2. 보기 선택
3. 제출 버튼 활성화
4. 제출 시 채점
5. 정답/오답 시각화
6. 해설 노출
7. 다음 문제 버튼 노출
8. 마지막 문제면 결과 화면으로 이동

### 6.3 결과 화면

이 화면은 세션의 종착점이면서 다음 학습 루프의 출발점이다.

필수 역할:

- 전체 성과 요약
- 맞춘 문제/틀린 문제 구분
- 오답만 다시 풀기 액션 제공
- 다른 모드 재도전 진입점 제공 가능

### 6.4 오답 복습 화면

두 가지 형태가 가능하다.

- 결과 화면 내부 필터
- 별도 `review` 라우트

문서는 둘 다 허용하지만, 제시된 폴더 구조상 별도 화면 분리가 더 자연스럽다.

### 6.5 대시보드 / 즐겨찾기 화면

이는 현재 세션보다 누적 기록을 보여주는 곳이다.

- 대시보드: 누적 정답률, 풀이 횟수, 최근 학습
- 즐겨찾기: 별표한 문제 목록과 재진입

즉, 실시간 문제풀이와 누적 학습 회고가 분리된다.

## 7. 상태 머신 관점에서 본 동작 방식

이 프로젝트는 UI보다 상태 전이 설계가 더 중요하다.  
문제를 한 개 기준으로 상태를 나누면 다음과 같다.

### 7.1 단일 문제 상태

- `idle`: 아직 보기 선택 전
- `selected`: 보기 선택 완료, 제출 전
- `submitted-correct`: 제출 완료, 정답
- `submitted-wrong`: 제출 완료, 오답

추가 파생 상태:

- `explanation-open`
- `favorited`

### 7.2 세션 상태

- `not-started`
- `in-progress`
- `completed`

세션 내부에 포함될 수 있는 값:

- 전체 문제 목록
- 현재 순서
- 현재 위치 인덱스
- 문제별 제출 결과
- 모드 정보
- 시작 시간 / 종료 시간(시험 모드 확장 시 유용)

### 7.3 중요한 상태 전이 규칙

문서가 직접 또는 암묵적으로 요구하는 규칙:

- 선택 전에는 제출할 수 없어야 한다.
- 제출 전에는 정답 표시가 없어야 한다.
- 제출 후에는 정답 여부와 해설을 볼 수 있어야 한다.
- 중복 제출 정책은 명확해야 한다.
- 마지막 문제 후에는 자연스럽게 결과 화면으로 전이되어야 한다.

중복 제출 정책은 문서에서 "명확히 처리"라고만 되어 있으므로, 구현 시 아래 둘 중 하나를 명시해야 한다.

- 제출 후 재선택/재제출 금지
- 제출 후 검토는 가능하지만 점수는 첫 제출만 인정

학습 앱 성격상 첫 번째가 더 단순하고 안전하다.

## 8. 데이터 모델 분석

문서는 두 개의 핵심 JSON 예시를 제공한다.

### 8.1 문제 데이터

예시 필드:

- `id`
- `question`
- `choices`
- `answer`
- `explanation`
- `category`

이 구조는 최소한의 퀴즈 엔진에 적합하다.  
특징은 다음과 같다.

- `id`는 문제 추적의 기준 키다.
- `question`은 텍스트 본문이다.
- `choices`는 배열이며, 현재는 단일 정답형 객관식에 맞춰져 있다.
- `answer`는 인덱스 기반이다.
- `explanation`은 제출 후 표시되는 학습 피드백이다.
- `category`는 필터링, 통계, 향후 과목별 분석 확장에 유용하다.

### 8.2 풀이 기록 데이터

예시 필드:

- `questionId`
- `selectedAnswer`
- `isCorrect`
- `isFavorite`
- `solvedAt`

이 구조는 "세션 결과"와 "영구 사용자 설정"이 일부 섞여 있다.  
특히 `isFavorite`가 풀이 기록 안에 있는 점은 주의가 필요하다.

해석 가능한 두 가지 방향:

1. 문제별 사용자 메타데이터를 하나의 레코드로 합친다.
2. 풀이 기록과 즐겨찾기를 분리 저장한다.

문서 예시는 1번에 가깝지만, 실제 확장성을 보면 2번이 더 안정적이다.  
이유는 다음과 같다.

- 즐겨찾기는 풀이 여부와 독립적인 상태다.
- 한 문제를 여러 번 풀면 기록이 누적될 수 있다.
- 이때 `isFavorite`를 각 풀이 레코드에 넣으면 중복과 불일치가 생긴다.

따라서 구현 관점에서는 다음처럼 나누는 것이 더 자연스럽다.

- 문제 마스터 데이터
- 현재 세션 상태
- 누적 풀이 이력
- 즐겨찾기 세트
- 최근 세션 요약

문서는 구조 변경 시 이유 설명을 요구하므로, 실제 구현 시 이런 분리는 반드시 설명과 함께 이루어져야 한다.

## 9. 저장소 설계 관점에서 본 LocalStorage 역할

문서에서 `LocalStorage`는 단순 캐시가 아니라 핵심 기능을 성립시키는 영속 계층이다.

저장 대상 후보:

- 문제별 최근 풀이 결과
- 누적 풀이 횟수
- 정오답 이력
- 즐겨찾기 목록
- 최근 학습 상태
- 현재 진행 중 세션 복원 정보

### 9.1 LocalStorage가 적합한 이유

- 개인용 앱이다.
- 서버를 두지 않는다.
- 정적 웹앱 우선이다.
- 브라우저 새로고침 이후 상태 유지가 필요하다.

### 9.2 LocalStorage의 한계

- 브라우저/기기 간 동기화 불가
- 구조 변경 시 마이그레이션 고려 필요
- 용량 제한 존재
- 데이터 무결성 보장이 약함

### 9.3 문서가 암묵적으로 요구하는 저장 전략

문서는 저장소를 깊게 설계하진 않았지만, 최소한 다음 원칙이 필요하다.

- 저장 키 네이밍 일관성
- 버전 관리 가능성
- 파싱 실패 시 안전한 기본값 복구
- 저장소 초기화 옵션 제공

즉, `lib/storage`는 단순 헬퍼가 아니라 앱 안정성을 좌우하는 계층이 된다.

---

## 10. 제안된 폴더 구조가 의미하는 아키텍처

문서가 제안하는 구조는 계층 분리를 꽤 명확하게 드러낸다.

### 10.1 `app/`

라우트 진입점이다.

예상 책임:

- 페이지 조합
- 화면 단위 데이터 로딩
- URL 기반 모드 분기
- 상위 레이아웃

예상 라우트:

- `/` 메인 화면
- `/quiz` 문제풀이
- `/result` 결과
- `/review` 오답 복습
- `/dashboard` 통계
- `/favorites` 즐겨찾기

### 10.2 `components/`

화면 조립용 UI 계층이다.

예상 하위 구조:

- `question/`: 문제 카드, 보기 리스트, 제출 컨트롤, 해설 패널
- `result/`: 요약 카드, 결과 목록, 점수 시각화
- `review/`: 오답 목록, 재도전 버튼
- `dashboard/`: 통계 위젯
- `common/`: 버튼, 레이아웃, 배지 등

### 10.3 `data/`

정적 문제 데이터 저장 위치다.

- `questions.json`이 시작점
- 추후 카테고리/과목별 파일 분리 가능

### 10.4 `lib/`

순수 로직과 타입, 저장 관련 기능이 집중되는 영역이다.

예상 책임:

- `storage/`: LocalStorage 입출력
- `quiz/`: 채점, 결과 계산, 랜덤화, 필터링
- `utils/`: 공통 유틸리티
- `types/`: 문제/기록/세션 타입

즉, 문서는 UI와 로직의 분리를 분명하게 요구하고 있다.

## 11. 핵심 로직 단위별 예상 책임

문서를 바탕으로 구현 시 반드시 필요해 보이는 로직 단위들을 정리하면 다음과 같다.

### 11.1 채점 로직

입력:

- 문제 정의
- 사용자가 선택한 보기 인덱스

출력:

- 정답 여부
- 정답 인덱스
- 선택 답안 인덱스

핵심 특징:

- 순수 함수로 분리하기 쉽다.
- 테스트가 쉬운 영역이다.
- 결과 화면 집계의 기초가 된다.

### 11.2 결과 집계 로직

입력:

- 세션 내 제출 결과 목록

출력:

- 총 문제 수
- 정답 수
- 오답 수
- 미응답 수
- 정답률
- 오답 문제 ID 목록

랜덤 모드와 시험 모드에서도 동일하게 재사용 가능해야 한다.

### 11.3 문제 세트 생성 로직

입력:

- 전체 문제 목록
- 모드 옵션
- 오답 문제 목록

출력:

- 실제 풀이에 사용할 문제 순서 목록

예시:

- 일반 모드: 원본 순서
- 랜덤 모드: 셔플된 순서
- 오답 복습 모드: 오답 문제만 추출
- 즐겨찾기 모드 확장 가능

### 11.4 저장/복원 로직

책임:

- 최근 세션 저장
- 통계 누적
- 즐겨찾기 저장
- 초기화 처리

이 부분은 화면 로직과 강하게 결합되기 쉬우므로 조심해서 추상화해야 한다.

## 12. 시험 모드와 학습 모드의 차이

문서에서 시험 모드는 상세 정의가 적지만, "학습 모드와 다른 UI/UX"를 허용한다는 점이 중요하다.

이를 바탕으로 보면 두 모드는 목표가 다르다.

### 12.1 학습 모드

- 즉시 피드백 중심
- 해설 확인이 자연스러움
- 학습 보조 기능 적극 노출 가능

### 12.2 시험 모드

- 연속 풀이 경험 중심
- 실제 시험처럼 몰입감 있는 흐름
- 결과 요약으로 자연스럽게 연결

문서상 최소 요구는 결과 화면까지 이어지는 자연스러운 흐름이다.  
즉, 시험 모드는 반드시 타이머가 있어야 하는 것은 아니며, 우선은 "연속 풀이 경험을 강조한 별도 흐름" 정도로 해석하는 것이 맞다.

---

## 13. 랜덤 모드가 단순 셔플 이상인 이유

문서의 랜덤 모드는 보기보다 중요하다.  
왜냐하면 단순히 순서만 바꾸는 기능이 아니라, 결과 계산과 복습 흐름이 정상 동작해야 하기 때문이다.

즉, 랜덤 모드 구현 시 보존해야 할 것은 다음이다.

- 문제의 정체성(`id`)
- 정오답 기록
- 최종 집계 일관성
- 오답 추출 정확성

문제 순서만 섞고 나머지 로직이 인덱스 기반으로 엉키면 결과 화면이 틀어질 수 있다.  
따라서 내부적으로는 "배열 위치"보다 "문제 ID"를 중심 키로 삼는 구조가 안전하다.

---

## 14. 즐겨찾기 기능의 위치와 의미

즐겨찾기는 부가 기능처럼 보이지만, 사실 오답 복습과는 다른 축의 개인화 기능이다.

- 오답 복습: 성과 기반 재학습
- 즐겨찾기: 사용자의 의도 기반 큐레이션

즉, 사용자는 틀리지 않았더라도 중요한 문제를 저장할 수 있어야 한다.  
이 때문에 즐겨찾기는 풀이 결과에 종속되기보다 문제 ID 기준의 독립 상태로 다루는 편이 더 적절하다.

예상 기능:

- 문제 화면에서 토글
- 즐겨찾기 목록 화면에서 확인
- 즐겨찾기 문제만 다시 풀기 확장 가능

---

## 15. 통계 대시보드가 요구하는 누적 데이터

문서가 제시한 통계 항목은 다음과 같다.

- 정답률
- 누적 풀이 수
- 오답 비율
- 즐겨찾기 수
- 최근 학습 정보

이 통계를 구현하려면 단순한 "현재 세션 점수"만으로는 부족하다.  
최소한 아래 데이터가 누적 저장되어야 한다.

- 세션별 또는 문제별 풀이 기록
- 최근 풀이 시각
- 총 제출 횟수
- 맞춘 횟수 / 틀린 횟수
- 즐겨찾기 개수

즉, 대시보드는 현재 세션 UI가 아니라 작은 학습 분석 계층이다.

---

## 16. 문서가 암묵적으로 유도하는 구현 우선순위

문서는 1차부터 5차까지 구현 단계를 제시한다.  
이 순서는 단순 일정표가 아니라 의존성 순서를 잘 반영하고 있다.

### 16.1 1차 구현

- 프로젝트 초기화
- 폴더 구조
- 타입 정의
- 샘플 데이터
- 메인 화면
- 문제풀이 화면

즉, "보이기 시작하는 최소 제품 골격"이다.

### 16.2 2차 구현

- 보기 선택
- 제출
- 정답 판정
- 해설
- 다음 문제
- 결과 화면

즉, "실제 퀴즈 엔진 완성" 단계다.

### 16.3 3차 구현

- 정답/오답 분리
- 오답 복습
- 다시 풀기

즉, 학습 앱으로서의 차별점이 붙는 구간이다.

### 16.4 4차 구현

- 랜덤 모드
- 시험 모드
- 즐겨찾기

기존 엔진에 변형 규칙과 개인화 기능을 얹는 단계다.

### 16.5 5차 구현

- 대시보드
- LocalStorage 저장/복원

문서 순서상 저장이 마지막으로 적혀 있지만, 실제 개발에서는 초기부터 저장 인터페이스를 염두에 두는 것이 좋다.  
그렇지 않으면 나중에 상태 구조를 크게 다시 손봐야 할 수 있다.

## 17. 완료 기준의 의미

완료 기준을 보면 문서는 "1차 완성 상태"라는 표현을 쓰지만, 실제 목록은 꽤 넓다.

완료로 간주되려면 다음이 전부 동작해야 한다.

- 문제풀이
- 보기 선택 및 제출
- 채점
- 해설
- 결과 화면
- 오답 복습
- 반응형
- 랜덤 모드
- 시험 모드
- 즐겨찾기
- 학습 통계
- LocalStorage 유지

즉, 문서상의 "완료"는 사실상 전체 MVP 완료에 가깝다.  
부분 기능만 있는 상태는 최종 완료로 볼 수 없다.

## 18. 협업 규칙 문서로서의 성격

이 문서는 단순 요구사항 문서를 넘어서 "Codex가 어떻게 일해야 하는가"까지 규정한다.

### 18.1 작업 프로토콜

코드 생성 전 반드시:

1. 목표 이해
2. 구현 계획 제시
3. 수정/생성 파일 목록 제시
4. 실제 작업

즉, 이 프로젝트에서는 코드를 바로 쓰는 것보다 계획과 변경 범위 설명이 먼저다.

### 18.2 범위 통제

금지 항목을 보면 문서는 특히 다음을 경계한다.

- 무관한 수정
- 설명 없는 구조 변경
- 한 번에 여러 핵심 기능 혼합
- 복잡한 추상화
- 불필요한 백엔드 도입

이 말은 곧, 이 프로젝트가 "작지만 통제된 코드베이스"를 지향한다는 뜻이다.

### 18.3 브랜치 전략

문서는 1인 개발임에도 브랜치 분리를 강하게 요구한다.

핵심 목적:

- 작업 단위 명확화
- 변경 범위 관리
- self-review 용이성
- 안정적인 `main` 유지

즉, 기술 문서이면서 동시에 작업 운영 문서다.

## 19. 아직 명시되지 않았지만 구현 시 반드시 결정해야 할 세부사항

문서가 방향은 잘 주지만, 실제 구현 전에 정해야 할 항목들도 있다.

### 19.1 제출 후 재수정 허용 여부

문서는 중복 제출 정책을 명확히 하라고만 말한다.  
가장 단순한 정책은 "제출 후 선택 변경 불가"다.

### 19.2 미응답 처리 기준

결과 요약에서 "필요 시 미응답 수"를 언급한다.  
학습 모드에서 모든 문제를 반드시 제출하도록 하면 미응답은 0일 수 있다.  
반면 시험 모드에 건너뛰기나 종료 기능이 들어가면 미응답 개념이 필요해진다.

### 19.3 세션 단위 기록 vs 문제 단위 누적 기록

통계와 최근 학습을 정확히 구현하려면 저장 구조를 어디까지 남길지 정해야 한다.

### 19.4 문제 분류 체계

예시에는 `category` 하나만 있지만, 실제 시험 대비 서비스라면 과목, 단원, 난이도 같은 확장이 곧 필요할 수 있다.

### 19.5 JSON 데이터 규모 증가 시 전략

지금은 `questions.json` 하나로 시작 가능하지만, 문제 수가 커지면 파일 분리나 지연 로딩이 필요할 수 있다.

## 20. 이 프로젝트가 실제로 "어떻게 동작해야 하는지" 한 문장씩 정리

아래는 문서를 구현 동작 관점으로 다시 압축한 것이다.

- 앱은 정적 문제 데이터(JSON)를 읽어 문제풀이 세션을 시작해야 한다.
- 사용자는 한 번에 한 문제를 보고 보기 하나를 선택해야 한다.
- 제출 전에는 채점 결과가 보이면 안 된다.
- 제출 직후 정답/오답과 해설이 보여야 한다.
- 문제풀이가 끝나면 결과 화면에서 성과를 요약해야 한다.
- 결과 화면은 오답 복습으로 이어져야 한다.
- 사용자는 랜덤 순서와 시험형 흐름으로도 문제를 풀 수 있어야 한다.
- 사용자는 문제를 즐겨찾기하고 다시 모아볼 수 있어야 한다.
- 앱은 학습 기록과 최근 상태를 브라우저에 저장해야 한다.
- 대시보드는 누적 기록을 바탕으로 학습 현황을 보여줘야 한다.

## 21. 종합 결론

현재 이 폴더는 구현체보다는 "구현 규칙 + 제품 요구사항 + 작업 방식"을 합쳐놓은 설계 기준점이다.  
따라서 이 프로젝트를 이해한다는 것은 단순히 기능 목록을 읽는 것이 아니라, 다음 구조를 이해하는 것이다.

- 정적 데이터 기반 문제풀이 앱
- 단일 문제와 전체 세션을 모두 관리하는 상태 중심 UI
- 복습과 개인화 기능까지 포함한 학습 루프
- LocalStorage를 영속 계층으로 사용하는 프론트엔드 중심 아키텍처
- 작은 단위 브랜치와 단계별 구현을 강제하는 운영 규칙

정리하면, 이 프로젝트는 "간단한 퀴즈 페이지"가 아니라 "로컬 상태 기반의 시험 대비 학습 시스템"으로 설계되어 있다.  
실제 구현 시 가장 중요한 성공 요인은 화려한 UI보다도 아래 세 가지다.

- 데이터 구조의 일관성
- 세션 상태 전이의 명확성
- 저장/복원 로직의 안정성

이 세 축이 안정되면 나머지 기능인 랜덤 모드, 시험 모드, 복습, 즐겨찾기, 통계는 같은 엔진 위에 자연스럽게 쌓을 수 있다.
