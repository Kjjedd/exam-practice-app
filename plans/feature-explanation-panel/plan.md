# feature/explanation-panel Plan

## Goal

제출이 끝난 현재 문제에 대해 해설과 정답/오답 피드백을 확인할 수 있는 해설 패널 UI를 만든다.

## Branch Intent

이 브랜치는 "제출 완료 상태 위에 해설을 얹되, 아직 다음 문제 이동이나 결과 화면으로는 넘어가지 않는다"는 목적만 담당한다.

- [x] 제출이 끝난 경우에만 해설 패널을 노출한다.
- [x] 사용자가 해설을 읽기 쉽게 접기/펼치기할 수 있는 UI를 만든다.
- [x] 정답/오답에 맞는 피드백 메시지를 해설 영역과 자연스럽게 연결한다.
- [x] 다음 단계의 문제 이동 로직이 자연스럽게 올라갈 수 있도록 해설 상태 책임을 분명히 유지한다.

이 단계는 "왜 정답인지 이해한다"는 경험을 만드는 구간이다.  
하지만 아직 다음 문제 버튼이나 세션 종료 흐름까지 포함하지는 않는다.

## Scope

- 제출 후 해설 패널 노출
- 해설 텍스트 표시
- 접기/펼치기 토글
- 정답/오답 피드백 메시지
- 제출 전 해설 비노출 처리
- 제출 후 결과와 해설의 정보 위계 정리

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- `feature/answer-checking`에서 만든 `isSubmitted`, `isCorrect` 상태를 해설 노출 조건으로 사용
- 현재 문제의 `explanation` 텍스트를 읽기 좋은 패널로 렌더링
- 해설 패널이 항상 펼쳐질지, 기본 접힘인지 정책을 정리
- 사용자가 필요 시 열고 닫을 수 있는 토글 UI 제공
- 정답/오답 피드백 문구와 해설 본문이 시각적으로 분리되되, 하나의 이해 흐름으로 읽히도록 구성

이번 브랜치의 핵심은 "판정 결과 뒤에 이해를 제공하는 것"이지, 문제 이동을 제어하는 것이 아니다.

## State Principles

해설 상태는 아래 원칙을 따라야 한다.

- 제출 전에는 해설이 절대 보이면 안 된다.
- 제출 후에만 해설 토글이 활성화되어야 한다.
- 해설 표시 상태와 제출 상태는 분리되어 있어야 한다.
- 해설 패널은 현재 문제 기준으로만 동작해야 한다.
- 이후 문제 이동이 생기면 새 문제에서 해설 패널 상태는 초기화될 수 있도록 단순한 구조를 유지한다.

## Proposed State Shape

이번 단계에서는 현재 문제에 대해 아래 수준의 상태가 적절하다.

- `isSubmitted: boolean`
- `isCorrect: boolean | null`
- `isExplanationOpen: boolean`

또는 이와 동등한 구조라면 가능하다.  
핵심은 해설 노출 여부가 제출 상태에 종속되어야 한다는 점이다.

## Interaction Design

이번 단계에서 기대하는 상호작용은 다음과 같다.

### 1. Before Submit

- 해설 패널 비노출
- 해설 토글 버튼 비노출 또는 비활성
- 정답/오답 피드백 문구도 비노출

### 2. Immediately After Submit

- 사용자는 정답/오답 결과를 본다.
- 해설 패널이 기본으로 열리거나, 열 수 있는 상태가 된다.
- 최소한 사용자가 해설을 볼 수 있다는 사실은 명확해야 한다.

### 3. Toggle

- 사용자는 해설을 접거나 다시 펼칠 수 있다.
- 해설 본문이 길어도 패널 구조가 무너지지 않아야 한다.

### 4. Non-goals in This Step

- 다음 문제 이동 버튼 없음
- 결과 화면 이동 없음
- 세션 요약 없음

## UI Principles

- 해설은 결과보다 약하지 않게, 하지만 결과를 덮어쓰지 않게 배치해야 한다.
- 정답/오답 피드백과 해설 본문은 서로 다른 층위의 정보로 보여야 한다.
- 해설 패널은 긴 텍스트에도 읽기 쉬워야 한다.
- 접힘 상태에서도 사용자가 "해설이 존재한다"는 사실을 알아야 한다.
- 해설 패널은 다음 단계에서 내비게이션 아래 혹은 위에 자연스럽게 배치될 수 있도록 독립 컴포넌트로 두는 편이 좋다.

## Explanation Display Rules

제출 후 해설 영역은 최소한 아래 정보를 다룰 수 있어야 한다.

- 정답/오답 요약 메시지
- 문제 해설 본문
- 해설 토글 상태

해설 본문은 `Question.explanation`을 그대로 쓰더라도, UI에서는 읽기 좋은 문단 블록처럼 보여야 한다.

## Feedback Message Rules

피드백 메시지는 최소한 아래 구분을 지원해야 한다.

- 정답인 경우: 정답임을 알려주고, 해설을 통해 왜 맞는지 보강
- 오답인 경우: 오답임을 알려주고, 해설을 통해 어디를 놓쳤는지 이해하게 도움

즉, 피드백은 결과를 반복하는 문장에 그치지 않고 해설 읽기로 자연스럽게 이어져야 한다.

## Data Usage Notes

이번 단계는 여전히 활성 문제 세트의 첫 문제를 기준으로 동작한다.

- 현재 문제의 `explanation`을 사용
- 현재 문제의 제출 상태를 해설 노출 조건으로 사용
- 활성 문제 세트가 없는 경우에는 기존 빈 상태를 그대로 유지

이번 브랜치에서는 문제별 해설 경험만 중요하며, 세션 전체 해설 모음은 다루지 않는다.

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- 해설 상태와 내비게이션 상태를 한 번에 섞으면 다음 브랜치 경계가 무너진다.
- 제출 전 해설이 보이면 문제풀이 의미가 크게 훼손된다.
- 결과 메시지와 해설 본문이 같은 시각 레벨로 섞이면 읽기 흐름이 혼란스러워진다.
- 해설 패널 상태를 과하게 복잡하게 만들면 다음 문제 이동 시 초기화가 어려워질 수 있다.

## Instant Trigger Revision

이후 수정에서는 해설 패널의 트리거 시점도 제출 버튼이 아니라 보기 클릭 즉시 판정 결과에 맞춰 바뀌어야 한다.

- 해설 패널은 더 이상 "제출 버튼을 누른 뒤" 열리는 구조가 아니다.
- 대신 사용자가 보기를 클릭해 현재 문제가 즉시 판정되는 순간, 현재의 제출 완료 상태와 같은 기준으로 해설 패널이 노출되어야 한다.
- 이미 판정된 문제를 다시 방문했을 때는 기존 결과 상태를 읽어 해설 노출 조건을 판단해야 한다.
