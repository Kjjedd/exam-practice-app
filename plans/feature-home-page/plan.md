# feature/home-page Plan

## Goal

사용자가 PDF 가져오기와 학습 시작 흐름을 모두 이해할 수 있는 메인 화면과 최소 네비게이션 구조를 만든다.

## Branch Intent

이 브랜치는 "프로젝트 부트스트랩용 임시 홈 화면을 PDF 업로드 기반 실제 제품의 시작 화면으로 교체한다"는 목적만 담당한다.

- [x] 사용자가 앱의 주요 흐름을 한눈에 이해할 수 있는 메인 화면을 만든다.
- [x] PDF 가져오기 진입점과 활성 문제 세트 상태를 명확히 보여준다.
- [x] 일반/랜덤/시험 모드 진입 조건을 활성 문제 세트 유무와 자연스럽게 연결한다.
- [x] 복습/즐겨찾기/통계 화면으로 이동할 수 있는 링크를 배치한다.
- [x] 아직 구현되지 않은 하위 기능은 홈 화면에서 자연스럽게 placeholder 경로로 연결한다.

이 단계에서는 홈 화면과 진입 흐름만 다루며, 실제 문제풀이 로직은 구현하지 않는다.  
기존의 "정적 샘플 데이터가 이미 있다"는 전제 대신, 현재 계획에서는 "활성 문제 세트가 없으면 먼저 가져오게 한다"는 흐름을 우선한다.

## Scope

- 홈 화면 레이아웃 구성
- PDF 가져오기 진입 UI
- 활성 문제 세트 상태 요약
- 일반 모드 진입 UI
- 랜덤 모드 진입 UI
- 시험 모드 진입 UI
- 복습/즐겨찾기/통계 화면으로 가는 진입 링크 구성
- 각 진입 요소의 최소 라우팅 연결 또는 placeholder 경로 준비
- 반응형 홈 화면 구조 구성

## Detailed Scope

이번 단계에서 다뤄야 하는 핵심은 아래와 같다.

- 현재 부트스트랩 전용 홈 화면을 제품 소개형 홈 화면으로 대체
- 앱의 핵심 목적과 PDF 기반 가져오기 흐름을 짧게 설명하는 상단 섹션 구성
- PDF 가져오기 CTA와 주요 모드 진입 카드 또는 버튼 구성
- 보조 진입 링크(복습/즐겨찾기/통계) 배치
- 이후 브랜치에서 연결이 자연스럽게 이어질 수 있는 정보 구조 정리

이번 브랜치의 핵심은 "시작 화면이 앱의 사용 흐름을 설명하고, 다음 행동을 제안하는 것"이다.

## UX Principles

홈 화면은 아래 원칙을 따라야 한다.

- 사용자가 처음 봐도 무엇을 하는 앱인지 바로 이해할 수 있어야 한다.
- 주요 시작 행동은 보조 링크보다 더 강조되어야 한다.
- 활성 문제 세트가 없을 때는 "가져오기"가 가장 중요한 행동이어야 한다.
- 일반 모드, 랜덤 모드, 시험 모드는 서로 구분되는 진입점이어야 한다.
- 복습, 즐겨찾기, 통계는 보조 기능이지만 홈에서 접근 가능해야 한다.
- 반응형 기준에서 모바일과 데스크톱 모두 자연스럽게 읽혀야 한다.

## Information Architecture

이번 단계에서 홈 화면에 포함되면 좋은 구조는 다음과 같다.

### 1. Hero Section

역할:

- 앱 소개
- 핵심 가치 전달
- 현재 사용자가 무엇을 해야 하는지 요약

예상 내용:

- 앱 이름 또는 제목
- 시험 대비/문제풀이 앱이라는 설명
- PDF 업로드 후 검수하여 문제 세트를 만드는 흐름 소개
- 학습 흐름에 대한 짧은 소개 문장

### 2. Primary Actions

역할:

- 핵심 시작 흐름 제공

포함 대상:

- PDF 가져오기 시작
- 일반 문제풀이 시작
- 랜덤 모드 시작
- 시험 모드 시작

고려사항:

- 가져오기 액션은 다른 학습 모드보다 우선순위가 높게 보일 수 있다.
- 문제 세트가 없을 때 시작 액션을 비활성 또는 안내 상태로 둘지 정리할 필요가 있다.
- 이후 실제 라우팅이나 모드 전달 방식이 바뀌더라도 홈 화면 구조는 유지 가능해야 한다.

### 3. Secondary Links

역할:

- 보조 학습 기능 진입

포함 대상:

- 오답 복습
- 즐겨찾기 보기
- 통계 대시보드 보기

고려사항:

- 아직 하위 기능이 완성되지 않았더라도 경로 진입 자체는 준비할 수 있다.
- placeholder 상태라도 홈 화면에서 정보 구조가 드러나는 것이 중요하다.

### 4. Active Set Summary

역할:

- 현재 활성 문제 세트의 존재 여부를 설명
- 사용자가 바로 학습 가능한 상태인지 알려준다

예시:

- 활성 문제 세트 이름
- 문제 수
- 마지막 가져오기 시각 또는 상태 안내

## Out of Scope

- 실제 문제풀이 동작
- PDF 추출 및 검수 상세 구현
- 문제 선택 상태 처리
- 채점/해설/결과 로직
- 저장/복원 로직
- 통계 계산 로직

- 실제 오답 복습 기능 구현
- 모드별 세부 동작 구현

## Routing Notes

현재 기준으로 홈 화면에서 연결 가능한 경로는 아래와 같다.

- 가져오기: `/import`
- 일반 문제풀이: `/quiz`
- 랜덤 모드: `/quiz` 기반 경로 또는 이후 확장 가능한 방식
- 시험 모드: `/quiz` 기반 경로 또는 이후 확장 가능한 방식
- 복습: `/review`
- 즐겨찾기: `/favorites`
- 통계: `/dashboard`

이번 단계에서는 실제 모드 동작보다 "사용자가 문제 세트를 가져오거나 해당 흐름으로 이동할 수 있는 진입점"이 더 중요하다.

## Dependency Notes

이 단계는 아래 브랜치들과 연결된다.

- `feature/question-data`에서 준비된 활성 문제 세트 상태를 홈 화면이 읽게 된다.
- `feature/pdf-upload-intake`와 `feature/pdf-import-review`는 홈 화면의 가져오기 진입 흐름과 직접 연결된다.
- `feature/question-ui`는 홈 화면에서 연결한 `/quiz` 흐름을 실제 문제 화면으로 발전시킨다.
- `feature/random-quiz-mode`, `feature/exam-mode`는 이번 홈 화면에서 만든 진입점을 나중에 실제 동작으로 채운다.
- `feature/wrong-answer-review`, `feature/favorites`, `feature/statistics-dashboard`는 이번에 노출된 보조 링크 흐름을 실제 기능으로 완성한다.

## Risks

이번 단계에서 특히 주의해야 할 위험은 다음과 같다.

- 홈 화면이 설명만 많고 실제 행동 유도가 약하면 시작 화면 역할을 못 하게 된다.
- 반대로 버튼만 많고 제품 설명이 없으면 앱의 목적이 흐려질 수 있다.
- 아직 구현되지 않은 하위 기능을 과하게 완성된 것처럼 보이게 만들면 사용 흐름이 어색해질 수 있다.
- 홈 화면 안에서 모드별 세부 동작까지 미리 확정하면 이후 브랜치에서 수정 비용이 커질 수 있다.
- 활성 문제 세트가 없는 상태를 무시하면 첫 사용자 경험이 끊어진다.

## Notes

- 홈 화면은 현재 앱의 진입 허브 역할에 집중한다.
- 제품 소개와 행동 유도 사이의 균형이 중요하다.
- 문제 세트가 없는 첫 진입 사용자를 위해 가져오기 흐름을 전면에 배치해야 한다.
- 이번 브랜치에서는 "제품다운 첫 화면"과 "가져오기 허브"를 함께 만드는 것이 핵심이다.

- 홈 화면에서 사용자는 현재 지원되는 시험 템플릿을 빠르게 이해할 수 있어야 한다.

이후 수정에서는 홈 화면에서 이어풀기 세션을 단순히 보여주는 것을 넘어서, 사용자가 그 세션을 직접 관리할 수 있어야 한다.

- 이어풀기 가능한 세션이 있으면 "이어 풀기" 외에도 "이어풀기 세션 삭제" 같은 정리 동작을 제공할 수 있어야 한다.

- 세션을 삭제하면 홈 화면 상태가 즉시 갱신되어 이어풀기 카드가 사라져야 한다.

## Range Picker Revision

첫 번째 신규 요구사항의 수정 방향은 "세트를 더 만드는 것"이 아니라, 현재 활성 세트 안에서 사용자가 직접 문제 범위를 지정해 학습을 시작할 수 있게 하는 것이다.

- 홈 화면은 여전히 활성 문제 세트 선택 UI를 제공하되, 선택한 세트에 맞는 문제 범위 입력 UI도 함께 제공해야 한다.
- 우선 범위 지정이 지원되는 세트는 `aws-saa-default`와 `aws-saa-600-plus`다.
- 사용자는 세트를 고른 뒤 시작 번호와 끝 번호를 입력하거나 선택할 수 있어야 한다.
- 예를 들어 `aws-saa-default`가 활성 세트일 때 `1~142`를 입력하면, 일반/랜덤/시험 모드가 모두 그 범위 안의 문제만 대상으로 시작해야 한다.
- 범위 입력 UI는 세트의 최소/최대 번호를 즉시 보여줘야 한다.
- 범위 입력이 비어 있거나 잘못되었을 때는 사용자가 바로 이해할 수 있는 안내가 있어야 한다.
- 범위 지정은 세트 전환과 별개로 관리되어야 하며, 세트를 바꾸면 그 세트의 기본 범위 또는 마지막 유효 범위로 갱신되는 편이 자연스럽다.
- 홈 화면의 학습 시작 CTA는 선택한 모드뿐 아니라 현재 세트와 범위를 함께 반영한 링크 또는 세션 시작 인자로 이어져야 한다.

이번 수정의 핵심은 "세트 선택 경험"을 "세트 + 범위 선택 경험"으로 확장하는 것이다.

### Range Picker Deliverables

- [x] 활성 세트 기준 최소/최대 번호 안내 UI
- [x] 시작 번호/끝 번호 입력 또는 선택 UI
- [x] 범위 유효성 검증 메시지
- [x] 일반/랜덤/시험 모드 진입에 범위를 함께 넘기는 시작 흐름
- [x] 세트 전환 시 범위 상태를 자연스럽게 갱신하는 UX

### Range Picker Validation

- [x] 홈 화면에서 활성 세트 기준 유효한 범위를 볼 수 있어야 한다.
- [x] 사용자가 범위를 지정한 뒤 일반/랜덤/시험 모드를 시작할 수 있어야 한다.
- [x] 잘못된 범위는 시작 전에 명확히 막혀야 한다.
- [x] 세트를 바꾸면 이전 세트의 범위가 그대로 남아 잘못 적용되면 안 된다.
- [x] 모바일과 데스크톱 모두에서 범위 입력 UI가 답답하지 않아야 한다.

## Set Switcher Cleanup Revision

추가 수정에서는 홈 화면의 세트 스위처를 더 단순하게 정리해야 한다.

- 세트 스위처에 보이는 기본 세트는 두 개만 남긴다.
- `aws-saa-default`
- `aws-saa-600-plus`
- 기존 임시 세트였던 `aws-saa-1-100`, `aws-saa-600-725`는 홈 스위처에서 제거되어야 한다.
- 세트 제목은 지금보다 더 빠르게 읽히는 짧고 분명한 라벨로 바꾸는 편이 좋다.
- 스위처 설명 문구도 "여러 임시 세트"를 전제로 하면 안 되고, 현재 유지하는 두 기본 세트와 범위 지정 흐름에 맞게 다시 써야 한다.
- 사용자는 "전체 세트"와 "600번 이후 세트"를 한눈에 구분할 수 있어야 한다.

### Set Switcher Cleanup Deliverables

- [x] 홈 화면에 두 기본 세트만 보여주는 스위처 UI
- [x] 더 읽기 쉬운 세트 제목 반영
- [x] 현재 구조에 맞는 간결한 스위처 안내 문구

### Set Switcher Cleanup Validation

- [x] 홈 화면에서 기본 세트가 두 개만 보여야 한다.
- [x] 각 세트 제목이 이전보다 짧고 직관적으로 읽혀야 한다.
- [x] 범위 지정 흐름과 스위처 설명 문구가 서로 어긋나면 안 된다.
