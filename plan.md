# ExamMate Development Plan

## 1. 문서 목적

이 문서는 [research.md](/Users/jongeon/Desktop/Study/ExamMate/research.md)를 바탕으로 ExamMate 프로젝트의 실제 개발 수행 계획을 정리한 문서다.  
목적은 다음과 같다.

- 구현해야 할 기능을 개발 가능한 단위로 분해한다.
- 각 기능에 대응하는 브랜치 전략을 명확히 한다.
- 기능 간 의존성을 정리해 안전한 개발 순서를 만든다.
- 각 단계에서 무엇을 만들고 무엇을 검증해야 하는지 기준을 제공한다.

이 문서는 코드 작성 전 계획 문서로 사용되며, 이후 실제 구현은 이 계획을 기준으로 진행하는 것을 전제로 한다.

---

## 2. 프로젝트 개발 원칙

본 계획은 기존 프로젝트 지침 문서와 `research.md`의 해석을 따른다.  
핵심 원칙은 다음과 같다.

- 하나의 브랜치는 하나의 작업 목적만 담당한다.
- 구현 순서는 의존성을 고려해 점진적으로 진행한다.
- UI와 로직, 데이터와 저장 계층을 분리한다.
- 데이터 구조 변경이 필요할 경우 이유를 먼저 설명한다.
- 기능 구현 후에는 타입, 흐름, 상태 일관성을 반드시 검증한다.
- 범위를 벗어나는 기능 추가는 하지 않는다.

---

## 3. 전체 개발 축

이 프로젝트는 크게 6개의 축으로 나뉜다.

1. 프로젝트 기반 구조
2. 문제 데이터 및 타입 체계
3. 문제풀이 핵심 엔진
4. 학습 보조 기능
5. 영속 저장 및 누적 기록
6. 통계 및 회고 화면

이 중 가장 중요한 축은 3번 문제풀이 핵심 엔진이며, 나머지 기능은 이 엔진 위에 쌓이는 구조다.

### 7.6 `feature/answer-selection`

#### 목표

사용자가 한 문제당 하나의 보기를 선택할 수 있게 만든다.

#### 주요 작업

- 단일 선택 상태 관리
- 선택한 보기 강조
- 제출 전 선택 변경 가능 처리
- 선택 안 했을 때 제출 비활성화 기반 준비

#### 완료 기준

- 문제당 하나의 보기만 선택된다.
- 선택한 보기가 시각적으로 구분된다.
- 제출 전에는 자유롭게 선택을 바꿀 수 있다.

#### 검증 포인트

- 중복 선택 방지
- 빠르게 클릭해도 상태가 꼬이지 않는지
- 선택 상태가 현재 문제와 정확히 연결되는지

### 7.7 `feature/answer-checking`

#### 목표

제출과 동시에 정답 여부를 판정하는 핵심 채점 로직을 만든다.

#### 주요 작업

- 제출 버튼 구현
- 선택 여부에 따른 제출 가능 제어
- 정답/오답 판정 함수 구현
- 제출 후 상태 고정
- 정답과 사용자 선택 답안 구분 표시

#### 완료 기준

- 보기 선택 후 제출할 수 있다.
- 제출 직후 정답 여부가 표시된다.
- 제출 전에는 채점 결과가 보이지 않는다.

#### 검증 포인트

- 채점 정확성
- 제출 후 재제출 제한 여부
- 마지막 문제 여부와 무관하게 결과 기록이 정상 반영되는지

---

- 최근 세션 저장
- 즐겨찾기 저장
- 새로고침 후 복원
- 초기화 옵션 추가

#### 저장 대상 후보

- 최근 세션 모드
- 문제별 최근 결과
- 누적 풀이 기록
- 즐겨찾기 목록
- 최근 학습 시각

#### 완료 기준

- 새로고침 후에도 학습 기록이 유지된다.
- 즐겨찾기 상태가 보존된다.
- 복원 실패 시 안전한 기본값으로 동작한다.

#### 검증 포인트

- 저장 파싱 오류 대응
- 비정상 데이터가 있을 때 복구 가능 여부
- 세션 데이터와 누적 데이터의 역할 분리가 명확한지

## 11. 리스크와 주의사항

개발 중 특히 주의해야 할 리스크는 다음과 같다.

### 11.1 인덱스 중심 설계 리스크

랜덤 모드, 오답 복습, 즐겨찾기 기능이 들어오면 배열 인덱스 중심 처리만으로는 결과가 어긋나기 쉽다.  
내부 추적은 가능하면 문제 ID 중심으로 유지해야 한다.

### 11.2 저장 구조 혼합 리스크

풀이 기록, 최근 세션, 즐겨찾기를 한 구조에 무리하게 섞으면 통계와 복원이 복잡해진다.  
역할을 분리해 저장하는 것이 안정적이다.

### 11.3 시험 모드와 학습 모드 충돌

모드별 상태를 명확히 분리하지 않으면 해설 노출 시점이나 결과 처리 방식이 충돌할 수 있다.

### 11.4 결과 집계 회귀 리스크

랜덤 모드, 오답 재풀이, 즐겨찾기 기반 재풀이 등이 추가되면 결과 집계 버그가 생기기 쉽다.  
결과 집계 로직은 가능한 순수 함수로 분리하는 것이 좋다.

## 13. 최종 정리

이 계획의 핵심은 기능을 한 번에 크게 구현하지 않고, 문제풀이 엔진을 중심으로 작은 단위 브랜치로 쪼개어 안정적으로 쌓아가는 것이다.  
프로젝트의 성공 여부는 화려한 확장 기능보다 아래 세 가지에 달려 있다.

- 문제 데이터 구조의 일관성
- 세션 상태 전이의 명확성
- LocalStorage 저장/복원 구조의 안정성

즉, 개발은 다음 순서의 철학을 따라야 한다.

1. 데이터와 타입을 고정한다.
2. 문제풀이 흐름을 완성한다.
3. 결과와 복습 흐름을 연결한다.
4. 모드 기능과 개인화 기능을 확장한다.
5. 저장과 통계를 안정화한다.

이 문서는 이후 실제 구현 브랜치를 열고 작업을 시작할 때 기준 계획서로 사용한다.
